# UX方法论框架使用规范

> 本文档定义11个UX方法论框架在高级UX总Skill工作流中的使用规则、衔接逻辑与约束边界。

---

## 第一部分：总览

### 高级UX总Skill 10步骤

| 步骤 | 名称 | 核心任务 |
|------|------|----------|
| ① | 原始需求识别与转译 | 把模糊需求翻译成清晰的设计问题 |
| ② | 设计目标定义 | 将问题转成可衡量的设计目标与非目标 |
| ③ | 用户与场景 | 定义用户画像、旅程地图、体验风险与页面优先级 |
| ④ | 假设梳理与验证 🆕 | 显式列出前提假设，评估风险与验证方式 |
| ⑤ | 设计机会识别 🆕 | 从功能缺口、体验痛点、风险转化三维度挖掘机会 |
| ⑥ | 整体交互链路设计 | 定义入口→主流程→弹窗→异常→成功态的完整路径 |
| ⑦ | 三套方案输出 | 完整呈现每套方案的链路、节点、优劣 |
| ⑧ | 异常流补全 | 覆盖取消、失败、超时、权限等全场景异常 |
| ⑨ | 最终方案定义 | 推荐方案+假设回检+理由+后续待确认事项 |
| ⑩ | 设计优先级与分期 🆕 | 将设计机会转化为可执行的分期交付计划 |

### 11框架分类与映射

| 分类 | 数量 | 框架名称 | 映射关系 |
|------|------|----------|----------|
| **融入型** | 3 | 5 Whys、How Might We、User Journey Map | 方法论已融入对应步骤执行逻辑，无需独立调用，执行该步骤时自动启用 |
| **正式使用型** | 2 | Assumption Validation Board (AVB)、Opportunity Solution Tree (OST) | AVB 在步骤④正式使用，OST 在步骤⑤正式使用，需输出标准产出物 |
| **融入+引用型** | 2 | Six Thinking Hats、Problem-Solution Fit | 六帽在步骤⑦⑨融入评审；PSF 在步骤⑨引用验证 |
| **内化型** | 1 | Design Thinking (IDEO) | 整个Skill底层方法论已内化其五步循环，无需单独执行 |
| **引用型** | 3 | User Interview 5Q、Competitive Analysis、User Story Mapping | 按需独立调用，在特定步骤作为辅助工具介入 |

### 分类使用原则

| 分类 | 使用时机 | 操作方式 | 是否必须 |
|------|----------|----------|----------|
| 融入型 | 执行对应步骤时自动启用 | 按步骤规范内化使用，不需单独输出框架文档 | 是，作为步骤执行标准 |
| 内化型 | 贯穿全程 | 不需要单独调用，Skill整体流程即其体现 | 是，作为底层思维模型 |
| 引用型 | 满足触发条件时按需调用 | 独立调用并输出该框架的标准产出物 | 否，按场景判断 |

### 各步骤输出呈现格式规范

每个步骤的产出必须使用规定的呈现格式，确保文档结构统一、可读性强、便于评审与交付。

#### 格式总览表

| 步骤 | 名称 | 主呈现格式 | 辅助格式 | 禁止格式 |
|------|------|-----------|----------|----------|
| ① | 原始需求识别与转译 | **结构化列表** + **5 Whys 追溯表** + **需求清晰度评分表** + **关键缺口清单** + **追问清单** | **Given-When-Then 行为表**、关键词标签 | 大段纯文字叙述、无置信度标注的判断 |
| ② | 设计目标定义 | **目标矩阵表** + **体验矛盾表** + **HMW 问题卡片** | OST 树形图（可选） | 无优先级标注的目标列表 |
| ③ | 用户与场景 | **用户旅程地图表格** + **体验风险清单** + **页面优先级初判表** | 用户画像卡片 | 纯文字场景描述 |
| ④ | 假设梳理与验证 🆕 | **假设分类总表**（用户/体验/业务） + **高风险假设聚焦表** | 假设状态标签（待验证/已验证/已推翻） | 无来源追溯的假设、无验证方式的判断 |
| ⑤ | 设计机会识别 🆕 | **设计机会总表**（功能/体验/风险转化） + **优先 Top 3-5 机会** + **机会→步骤映射表** | 机会优先级矩阵 | 无上游依据的机会、悬空不承接的机会 |
| ⑥ | 整体交互链路 | **流程路径表** + **页面三件套表格** + **状态机表** + **信息架构实体表** + **关键断点与优化节点表** + **全局交互规范表** | **低保真线框图**（核心页面） | 无节点编号的自由叙述 |
| ⑦ | 三套设计方案 | **方案卡片 × 3** + **三方案对比矩阵表** | **低保真线框图**（关键节点差异对比） | 三方案混在一起描述 |
| ⑧ | 异常流补充 | **异常流矩阵表** | 异常严重度分级标签 | 遗漏常见异常场景 |
| ⑨ | 推荐方案建议 | **推荐决策卡片** + **六帽评审表** + **假设回检表** | PSF 匹配度评分表 | 只给结论不给论据、未回检假设 |
| ⑩ | 设计优先级与分期 🆕 | **优先级排序总览表** + **分期交付计划表** + **待确认决策表** | Story Map 拆解 | 无分期依据的排序、无验证标准的交付 |

> **说明**：数据埋点方案、竞品对标总结、设计原则、下一步行动等模块归入步骤⑨的推荐方案建议中，作为最终方案的支撑材料。**状态机表**和**低保真线框图**是两个贯穿步骤⑥⑦的核心呈现格式，下面有独立模板和规范。

#### 各步骤格式详细说明

**① 原始需求识别与转译**

| 子项 | 格式要求 | 示例结构 |
|------|----------|----------|
| 原始诉求 | 单行引用块（> 格式） | > "我想做一个XXX功能" |
| 需求理解清单 | **结构化表格**，每项含：维度/当前理解/类型（事实/推断）/依据/**置信度** | 见下方模板 |
| 需求清晰度评分 | **5维度评分表**，每维度1-5分 + 缺口 + 置信度 | 见下方模板 |
| 需求拆解 | 结构化列表，每项含：需求点 / 判断依据 / 信息充分度标记（✅充分 / ⚠️待确认 / ❌缺失） | - 需求点：XXX ✅ |
| 5 Whys 根因追溯 | **纵向追溯表格**，每行一层 Why，最后一行为根因 | 见下方模板 |
| 功能行为描述 | **Given-When-Then 表格**，精确描述功能行为 | 见下方模板 |
| 需求转译结论 | 一句话定义 + 问题类型标签（功能型/流程型/体验型/信息型） | 转译：「用户需要在XX场景下完成YY操作」[流程型] |
| 下一步判断 | **判断摘要表**：判断结果/依据/关键假设/不确定性/待确认事项/建议 | 见下方模板 |
| 关键缺口清单 | **优先级排序表**：优先级/缺口/影响的用户行为/不补齐的风险/补全建议/置信度 | 见下方模板 |
| 追问清单 | **追问管理表**：优先级/追问问题/阻塞对象/若暂不确认的处理方式 | 见下方模板 |

需求理解清单模板：

> 💡 **填写要求**：至少覆盖以下维度：产品形态、核心任务、用户对象、输入方式、输出规格、使用场景、技术约束、成功标准。每行必须标注"类型"（事实/推断）和"置信度"（高/中/低）。事实=需求原文或官方文档明确说明；推断=基于竞品或经验合理推导。置信度低于"中"的维度需在关键缺口清单中标注。

**模板格式**：
```
| # | 需求要素 | 理解转译 | 置信度 | 依据 |
|---|---------|---------|--------|------|
| 1 | [需求原文关键词] | [转译后的具体理解] | [高/中/低] | [来源说明] |
```

**填写示例（以AI视频爆款复刻为例）**：
```
| # | 需求要素 | 理解转译 | 置信度 | 依据 |
|---|---------|---------|--------|------|
| 1 | AI 生成视频 | 用户通过文本/图片/参考视频输入，AI 生成视频内容 | 高 | 需求原文明确 |
| 2 | 爆款复刻 | 发现热门视频 → AI 拆解结构 → 用户替换内容 → 一键生成同款 | 高 | 蝉镜官网"爆款跟创"功能描述 |
| 3 | Web 工具形态 | 浏览器端使用，无需安装客户端 | 高 | 需求原文明确 |
| 4 | 爆款视频来源 | 抖音/快手/小红书等平台公开视频链接 | 中 | 凡科爆款复刻支持平台 |
| 5 | 复刻精度要求 | 保留原视频风格/结构/节奏，替换内容/主体/素材 | 中 | Kickart 爆款裂变"高吻合度"描述 |
| 6 | 目标用户 | 短视频创作者/电商运营/MCN 机构/知识博主 | 中 | 蝉镜用户画像 |
| 7 | 生成视频时长 | 15-60 秒短视频为主 | 中 | 可灵/即梦等工具时长范围推断 |
| 8 | 批量生成能力 | 单次任务可生成多条复刻视频 | 低 | Kickart 支持一次 100 条 |
```

需求清晰度评分模板：

> 💡 **填写要求**：必须覆盖5个维度：产品形态、主流程、用户定位、输出标准、体验标准。每个维度1-5分，≤2分的维度必须在后续步骤中优先补齐。综合分为5个维度的算术平均值。综合分≤2.0时，需在"下一步判断"中标注"需求高度模糊，需大量追问补全"。

**模板格式**：
```
| 维度 | 评分(1-5) | 说明 |
|------|----------|------|
| [维度名] | [X/5] | [评分理由和缺口说明] |
| **综合** | **[X.X/5]** | [总体判断] |
```

**填写示例（以AI视频爆款复刻为例）**：
```
| 维度 | 评分(1-5) | 说明 |
|------|----------|------|
| 用户目标 | 3 | "爆款复刻"方向明确，但复刻深度（结构级/画面级/情绪级）未定义 |
| 功能边界 | 2 | 缺少输入方式、输出规格、支持平台等边界说明 |
| 使用场景 | 2 | 仅知 Web 端，具体场景（追热点/批量投放/内容获客）未明确 |
| 技术约束 | 1 | 未说明 AI 生成能力边界、视频时长上限、并发限制 |
| 成功标准 | 1 | 无任何可衡量的成功指标定义 |
| **综合** | **1.8/5** | 需求高度模糊，需大量追问补全 |
```

Given-When-Then 行为表模板：

> 💡 **填写要求**：每个核心功能至少对应1条 Given-When-Then 描述。功能按主流程顺序排列，覆盖从入口到交付的完整行为链。每行标注置信度。Given 描述前提条件（用户状态+页面状态），When 描述用户的具体动作，Then 描述系统的明确反馈。

**模板格式**：
```
| # | 功能 | Given | When | Then | 置信度 |
|---|------|-------|------|------|--------|
| G[N] | [功能名称] | [前提条件] | [用户动作] | [系统反馈] | [高/中/低] |
```

**填写示例（以AI视频爆款复刻为例）**：
```
| # | 功能 | Given | When | Then | 置信度 |
|---|------|-------|------|------|--------|
| G1 | 链接导入 | 用户已登录，在爆款复刻页面 | 用户粘贴抖音/快手视频链接并提交 | 系统解析视频，展示结构化拆解结果 | 高 |
| G2 | 视频拆解 | 系统已成功导入爆款视频 | 拆解完成 | 展示分镜表（时间轴/场景/脚本/拍摄技巧/情绪标签） | 高 |
| G3 | 脚本定制 | 用户已查看拆解结果 | 用户填写行业/商品/卖点等信息并提交 | AI 基于原结构生成定制化脚本 | 高 |
| G4 | 素材替换 | 用户已确认定制脚本 | 用户上传素材图片/视频/数字人 | 系统将素材匹配到对应分镜槽位 | 中 |
| G5 | 一键生成 | 用户已完成素材替换 | 用户点击"生成视频" | 系统调用 AI 模型生成复刻视频，展示预览 | 高 |
| G6 | 模板浏览 | 用户在模板广场页面 | 用户浏览/搜索/筛选模板 | 展示模板卡片（预览/热度/行业标签） | 中 |
| G7 | 自由创作 | 用户在创作页面 | 用户输入文本/上传图片 | AI 生成视频，支持参数调节 | 高 |
| G8 | 批量生成 | 用户已配置批量任务 | 用户点击"批量生成" | 系统按矩阵组合生成多条视频 | 低 |
```

> 💡 **置信度标注规则**：每条判断必须标注置信度（高/中/低）。高=有事实依据或数据支撑，中=合理推断但未验证，低=经验直觉或间接证据。禁止出现无置信度标注的判断。

下一步判断模板：

> 💡 **填写要求**：基于需求清晰度评分和关键缺口清单综合判断。"判断结果"必须明确回答"需求是否清晰可进入设计"。"最大风险"需从缺口中提炼最可能导致方向偏差的一项。"是否可以并行推进"回答能否在追问的同时基于假设先行设计。

**填写示例（以AI视频爆款复刻为例）**：
```
| 判断项 | 结论 | 置信度 |
|--------|------|--------|
| 需求是否清晰可进入设计？ | 否，综合清晰度 1.8/5，关键缺口 8 项 | 高 |
| 最大风险 | 功能边界模糊，可能做出来与用户预期不一致 | 高 |
| 建议下一步 | 基于竞品调研和合理假设补全缺口，产出完整 UX 方案供评审 | 高 |
| 是否可以并行推进 | 是，可用"假设先行"策略推进设计，同时追问确认 | 中 |
```

5 Whys 追溯表模板：

> 💡 **填写要求**：追问至根因必须在设计能力可干预范围内。最后一行必须产出"核心结论"（用 ✅ 标注），总结根因并指向产品方向。每层的"回答"必须是下一层"问题"的直接原因，因果链必须可反向验证。

**填写示例（以AI视频爆款复刻为例）**：
```
| 层级 | 问题 | 回答 |
|------|------|------|
| Why 1 | 为什么需要爆款复刻功能？ | 因为用户缺乏创作方法论，不知道爆款为什么火 |
| Why 2 | 为什么不知道爆款为什么火？ | 因为拆解视频结构需要专业经验和大量时间 |
| Why 3 | 为什么拆解需要专业经验？ | 因为爆款涉及分镜节奏、情绪曲线、钩子设计等隐性知识 |
| Why 4 | 为什么隐性知识难以获取？ | 因为传统工具只提供剪辑能力，不提供创作策略分析 |
| Why 5 | 根因：用户真正需要什么？ | **不是"抄视频"，而是将已被验证的爆款结构迁移到自己的内容上**，降低创作门槛、提高爆款概率 |

> ✅ **核心结论**：爆款复刻的本质是"结构提取 + 素材重组"，而非简单搬运。产品应聚焦于**降低从"看到爆款"到"发布同款"的时间差和技能门槛**。
```

关键缺口清单模板：

> 💡 **填写要求**：至少覆盖5-6个缺口，按优先级排序（P0/P1/P2）。每个缺口必须明确"影响的用户行为"和"不补齐的风险"，以便评估紧急程度。"建议处理"方式包括：需追问确认、建议内置、可作二期等。P0缺口必须在进入步骤②前处理或有明确降级策略。

**填写示例（以AI视频爆款复刻为例）**：
```
| # | 缺口描述 | 影响范围 | 紧急度 | 建议处理 |
|---|---------|---------|--------|---------|
| 1 | 爆款视频来源范围未定义（支持哪些平台？是否支持本地上传？） | 核心流程 | 高 | 需追问确认 |
| 2 | AI 拆解的粒度未定义（仅脚本级？还是画面级/情绪级？） | 核心体验 | 高 | 需追问确认 |
| 3 | 复刻视频的时长/分辨率/比例规格未定义 | 输出质量 | 高 | 需追问确认 |
| 4 | 数字人形象是否参与复刻流程未明确 | 功能边界 | 中 | 需追问确认 |
| 5 | 是否需要版权保护/内容审核机制 | 合规风险 | 中 | 建议内置 |
| 6 | 批量生成的上限与计费规则未定义 | 商业模式 | 中 | 需追问确认 |
| 7 | 是否支持跨行业复刻（3C 爆款→家居） | 功能深度 | 低 | 可作二期 |
```

追问清单模板：

> 💡 **填写要求**：至少覆盖5-6个追问项，按优先级排序（P0/P1/P2）。每个追问必须明确"阻塞对象"（即该问题不回答会导致哪个设计环节无法推进）和"若暂不确认的处理方式"（降级策略）。追问清单的核心不是"等答案再开工"，而是明确每个缺口的**降级策略**，让设计可以在不确定性中持续推进。

```
| Q5 | 一期是否需要批量生成？还是先做单条复刻？ | 确定 MVP 范围 | P1 |
| Q6 | 是否需要视频发布/分发能力？ | 确定产品边界 | P1 |
| Q7 | 用户是否需要保存/管理拆解模板？ | 确定模板管理需求 | P2 |
```

> 💡 **填写要求**：至少识别4-5对体验矛盾。每对矛盾必须明确"倾向"（偏效率/偏控制/平衡等）和"处理策略"（具体的设计手段）。体验矛盾不是要"二选一"，而是帮设计师看清张力所在，在方案中做出有意识的平衡取舍。矛盾来源于用户调研、竞品分析或5 Whys根因分析中的发现。

```
|---|--------|---------|------|---------|--------|

```

**填写示例（以AI视频爆款复刻为例）**：
```
| # | 矛盾对 | 矛盾描述 | 倾向 | 处理策略 | 置信度 |
|---|--------|---------|------|---------|--------|

```

```
| Toast | 只用于轻反馈，3s自动消失，不承载复杂错误 | "脚本已保存""素材已匹配" | 生成失败等复杂错误需用页面内提示区+重试按钮 |
| 空状态 | 说明当前无内容并给主操作引导 | 模板广场无搜索结果→"未找到相关模板，试试换个关键词或自由创作" | 权限空态需先解释限制再引导升级 |
| 网络异常 | 明确失败原因+保留用户输入+提供重试 | 素材上传中断→保留已填信息+红色"上传失败"按钮 | 文件损坏不归为网络异常，需单独提示格式问题 |
| 弹窗规范 | 只用于确认高影响操作或必须中断的流程 | P5合规提醒弹窗：必须确认后才能导出 | 普通参数修改用内联编辑不用弹窗 |
| 表单规范 | 默认值优先+错误就近提示+实时校验 | 视频比例默认9:16，商品名填写后AI自动提取卖点 | 高级参数（风格强度等）折叠在"更多设置"中 |
| 返回/导航 | 返回不丢失任务上下文，生成中返回任务仍继续 | 从预览页返回脚本定制页，已填信息保留 | 刷新页面时弹出"未保存内容将丢失"确认 |
```

#### 状态机规范

> 状态机是步骤④中定义页面/组件动态行为的核心呈现格式。每个有 ≥ 2 种状态的页面或组件，都必须输出状态机，确保开发能准确实现每种状态及其切换逻辑。

**使用场景与必选规则**

| 场景 | 是否必选 | 说明 |
|------|----------|------|
| 列表页（空态/有数据/加载中/错误） | ✅ 必选 | 几乎所有列表都有多种状态 |
| 表单页（初始/编辑中/提交中/成功/失败） | ✅ 必选 | 表单交互必有状态变化 |
| 按钮/操作组件（默认/悬停/禁用/加载中） | ✅ 必选 | 交互组件必有状态机 |
| 弹窗/抽屉（未触发/已弹出/操作中/已关闭） | ✅ 必选 | 弹窗的生命周期 |
| 纯静态展示页（如关于页） | ❌ 不需要 | 无状态变化的页面不输出 |

**状态机输出结构**（每个页面/组件一个完整状态机）：

| 组成部分 | 内容 | 格式 |
|----------|------|------|
| **状态定义表** | 枚举所有状态，含状态ID、名称、触发条件、页面表现、可执行操作 | 表格 |
| **状态迁移图** | ASCII 文本形式展示状态间的迁移路径和触发事件 | 代码块 |
| **状态迁移规则表** | 每条迁移的当前状态、触发事件、目标状态、迁移条件、过渡动画、备注 | 表格 |

**状态编号规则**：
- 统一使用 `ST0` 起始，按逻辑顺序递增（ST0, ST1, ST2...）
- ST0 固定为初始/空态
- 异常状态统一放最后（如 ST_error, ST_timeout）

**完整性自检**：

| 检查项 | 要求 |
|--------|------|
| 状态覆盖 | 至少覆盖：初始态、加载态、成功态、失败/错误态、空态 |
| 迁移闭合 | 每个状态至少有一条迁出路径，不允许出现"死状态" |
| 异常覆盖 | 网络异常、超时、权限不足等异常状态必须有对应迁移 |
| 用户可操作 | 每个状态必须明确列出用户可执行的操作 |
| 编号联动 | 低保真线框图注释中引用的状态，必须与状态机 ST 编号一致 |

## 1. Design Thinking (IDEO)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 以人为中心的创新问题解决方法论，通过共情→定义→构思→原型→测试五步循环驱动设计决策 |
| 所属阶段 | 贯穿全流程（发现/定义/方案/验证/落地） |
| 与10步骤关系 | **内化于整个Skill** — 10步骤工作流的底层逻辑即Design Thinking五步循环的落地实现 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 执行任何UX总Skill任务 | ✅ 始终生效，无需单独调用 |
| 需求明确、仅做微调 | ⚠️ 可简化Empathize环节，但仍遵循Define→Ideate→Prototype→Test逻辑 |

### 与10步骤的对应关系

| Design Thinking步骤 | 对应Skill步骤 | 体现方式 |
|---------------------|---------------|----------|
| Empathize（共情） | ①③ 需求识别与转译 + 用户与场景 | 理解用户真实处境，追溯需求根因，构建用户画像与旅程 |
| Define（定义） | ②④ 设计目标定义 + 假设验证 | 将洞察转化为明确的设计问题和目标，显式列出前提假设 |
| Ideate（构思） | ⑤⑥⑦ 设计机会 + 链路设计 + 三方案 | 发散机会空间，设计多种交互链路和方案可能 |
| Prototype（原型） | ⑧⑨ 异常补全 + 推荐方案 | 将方案完整描述为可评审的交互定义，含异常处理 |
| Test（验证） | ⑨⑩ 假设回检 + 分期规划 | 通过假设回检和分期规划完成验证与迭代计划 |

### 操作步骤

不需要独立操作步骤。整个UX总Skill的执行过程即为Design Thinking的实践。

### 输出规范

不单独产出Design Thinking文档，其成果体现在10步骤的完整输出中。

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止跳过共情直接定义 | 即使需求看似清晰，也要在步骤①中追问"为什么" |
| 禁止线性执行 | 五步是循环而非瀑布，后续步骤发现问题需回退修正 |
| 禁止只关注方案不关注问题 | Define阶段的质量决定后续所有方案的上限 |

## 2. Opportunity Solution Tree (Teresa Torres)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 以目标为起点的树形结构化工具，将机会、方案、实验三层关联，确保每个方案都指向可验证的目标 |
| 所属阶段 | 机会识别阶段（步骤⑤正式使用，步骤②③④可作为输入） |
| 与10步骤关系 | **正式使用在步骤⑤（设计机会识别）**，是该步骤的核心方法论工具 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 需求涉及多目标、多用户群体 | ✅ 必须使用，避免方案散乱 |
| 方案方向不确定、需结构化探索 | ✅ 必须使用 |
| 简单单点优化需求 | ❌ 不需要，直接使用步骤④的三方案模式 |
| 已有明确方案、仅需执行 | ❌ 不需要 |

**使用前置条件**：步骤①~④已完成，设计目标、用户场景、假设清单已明确。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 明确的设计目标/业务目标 | 步骤②产出 |
| 用户洞察/痛点列表 | 步骤①③产出 |
| 已知约束条件 | 步骤②非目标定义 |
| 假设清单 | 步骤④产出 |

### 操作步骤

1. **定义目标**：将步骤②的设计目标写入树的根节点
2. **识别机会**：从步骤③的用户洞察和步骤④的假设中提取所有可能的改进机会，按功能/体验/风险转化分类，作为第二层节点
3. **发散方案**：每个机会至少生成3个方案思路，作为第三层节点
4. **设计实验**：为每个方案设计最小验证实验（用户访谈/原型测试/数据验证）
5. **执行裁剪**：根据实验结果标记置信度，裁剪低置信分支，保留高置信路径

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| OST树形图 | 缩进列表 或 结构图 | 目标→机会→方案→实验，每个节点带置信度（高/中/低） |

**质量标准**：
- 每个机会节点至少3个方案
- 每个方案至少1个可执行实验
- 节点间因果关系可追溯

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 5 Whys（步骤①）| 根因分析产出作为机会识别输入 |
| 上游 | HMW（步骤②）| HMW问题作为机会节点 |
| 上游 | 假设清单（步骤④）| 高风险假设的验证结果指导机会优先级 |
| 下游 | 步骤⑥ 交互链路设计 | 高置信机会进入正式链路设计 |
| 下游 | 步骤⑦ 三方案设计 | 高置信方案进入正式三方案详细设计 |
| 下游 | 步骤⑩ 设计优先级与分期 | 机会→步骤映射进入分期计划 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止跳过机会层直接列方案 | 方案必须挂载在机会节点下，不能悬空 |
| 禁止每个机会只有1个方案 | 至少3个方案才能形成有效比较 |
| 禁止不设计实验 | 没有实验的方案只是假设，不能进入下一步 |

## 3. 5 Whys

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 通过连续追问"为什么"逐层下钻，从表面诉求追溯到系统性根因的分析方法 |
| 所属阶段 | 需求理解阶段（步骤①） |
| 与10步骤关系 | **融入步骤①**，作为需求转译的子方法，在需求模糊或表面诉求可能不是真正问题时启用 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 需求来源模糊（如"老板说要做XX"） | ✅ 必须使用 |
| 需求带有预设方案（如"加个按钮"） | ✅ 必须使用 |
| 需求已经过清晰定义且因果明确 | ❌ 不需要 |
| 紧急执行型需求 | ⚠️ 可简化为2-3层追问 |

**使用前置条件**：已有原始需求输入（PRD/口头要求/反馈等）。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 原始诉求/问题陈述 | 需求输入（PRD、飞书消息、会议记录、用户反馈等） |

### 操作步骤

1. **陈述问题**：将原始诉求写成一个明确的问题陈述
2. **问Why 1**：为什么会出现这个问题？→ 第一层原因
3. **逐层追问**：每层基于上一层因果逻辑追问，直到触达结构性根因（通常3-5层）
4. **反向验证**：从根因反向推导，确认每层因果关系成立
5. **识别根因与对策方向**：明确真正根因，指向初步对策方向

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 因果链 | 树形缩进文本 | 问题陈述、Why 1-5、根因总结、对策方向 |

**输出模板**：

```
问题陈述：[原始诉求]
├─ Why 1：[第一层原因]
│  └─ Why 2：[第二层原因]
│     └─ Why 3 → ... → Why N → 根因
根因：[总结]
对策方向：[基于根因的对策]
```

**质量标准**：每层因果关系必须可验证，禁止跳跃推理。

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 原始需求输入 | 步骤①的需求素材 |
| 下游 | HMW（步骤②）| 根因/对策方向转化为HMW问题 |
| 下游 | OST（引用）| 根因作为机会识别的输入 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止机械追问5次不问逻辑 | 关键是因果链成立，不是次数 |
| 禁止跳到无法行动的根因 | 根因必须在设计能力可干预范围内 |
| 禁止忽略反向验证 | 必须从根因反推验证因果链完整性 |

## 4. How Might We (HMW)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 将用户洞察和根因转化为开放式设计问题的转译工具，用"我们如何能……"句式打开方案空间 |
| 所属阶段 | 问题定义阶段（步骤②） |
| 与10步骤关系 | **融入步骤②**，作为洞察→设计问题的转译工具 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 步骤①产出了多个洞察/痛点 | ✅ 必须使用，将洞察转为可设计的问题 |
| 需求目标明确但路径不清 | ✅ 使用，拓展方案空间 |
| 已有明确PRD指定具体方案 | ❌ 不需要，直接按PRD执行 |

**使用前置条件**：步骤①已完成，至少有2-3个用户洞察或根因。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 用户洞察/痛点列表 | 步骤①产出（5 Whys根因、User Interview洞察） |
| 设计目标 | 步骤②初步定义 |

### 操作步骤

1. **从洞察出发**：逐个审视步骤①的用户洞察
2. **写HMW问题**：将每个洞察转为"我们如何能……"句式
3. **检查范围**：确认问题范围不过大（无法下手）也不过小（限制方案）
4. **生成变体**：每个洞察生成3-5个HMW问题变体
5. **选出1-2个**：按影响力和可设计性筛选，进入步骤③链路设计

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| HMW问题集 | 列表 | 来源洞察、HMW问题、范围评估（过大/合适/过小）、入选标记 |

**质量标准**：
- 每个洞察至少3个HMW变体
- 最终选出1-2个进入链路设计
- 问题必须是开放式的，不能隐含预设方案

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 5 Whys（步骤①）| 根因和对策方向 |
| 上游 | User Interview 5Q（引用）| 访谈洞察 |
| 下游 | 步骤③ 链路设计 | 入选HMW问题驱动交互链路方向 |
| 下游 | OST（引用）| HMW问题作为机会节点 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止HMW中暗含方案 | ❌ "我们如何能加一个按钮" → ✅ "我们如何能让用户更快完成确认" |
| 禁止范围过大 | "我们如何能提升用户体验" 过于空泛 |
| 禁止不做筛选全部进入设计 | 必须收敛，否则会发散失控 |

## 5. Assumption Validation Board

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 系统化梳理和验证方案中隐含假设的看板工具，按风险优先级安排验证实验 |
| 所属阶段 | 假设验证阶段（步骤④正式使用，步骤⑨回检） |
| 与10步骤关系 | **正式使用在步骤④（假设梳理与验证）**，用于在进入设计前显式列出所有前提假设；步骤⑨推荐方案时回检假设状态 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 方案包含多个未验证的业务假设 | ✅ 必须使用 |
| 方案涉及技术可行性不确定 | ✅ 必须使用 |
| 方案基于成熟模式、假设已充分验证 | ❌ 不需要 |
| 时间极紧、只能先上线后验证 | ⚠️ 记录假设，延后验证 |

**使用前置条件**：步骤①~③已完成，需求、目标、用户场景已明确，从中可提取隐含假设。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 需求理解与缺口清单 | 步骤①产出 |
| 设计目标与非目标 | 步骤②产出 |
| 用户旅程痛点与体验风险 | 步骤③产出 |
| 业务/技术/用户行为假设 | 上述步骤中识别的隐含前提 |

### 操作步骤

1. **列出假设**：从步骤①~③的产出中提取所有隐含假设（需求理解中的推断、目标中的前提、用户场景中的预设）
2. **分类标注**：按用户假设/体验假设/业务假设分类，追溯每条假设的来源（步骤①/②/③的哪个结论）
3. **风险排序**：按"影响大 × 不确定性高"排优先级，标注每条假设不成立的影响范围
4. **设计实验**：为Top 3-5高风险假设设计最小验证实验
5. **执行记录**：记录实验结果，更新假设状态（已验证/已推翻/待验证）
6. **步骤⑨回检**：在推荐方案时回检假设清单，确认推荐方案是否依赖于未验证的高风险假设

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 假设分类总表 | 表格 | 假设描述、分类(用户/体验/业务)、来源追溯、不成立的影响、验证方式、置信度、当前状态 |
| 高风险假设聚焦表 | 表格 | 高风险假设、置信度、降级策略、触发验证的时间节点 |

**质量标准**：
- 所有高优先级假设必须有验证计划
- 已推翻的假设必须回溯影响对应方案

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 步骤① 需求转译 | 需求理解清单中的推断和假设 |
| 上游 | 步骤② 目标定义 | 设计目标中隐含的前提 |
| 上游 | 步骤③ 用户与场景 | 用户旅程中的痛点假设和场景预设 |
| 下游 | 步骤⑤ 设计机会识别 | 假设验证结果指导机会优先级排序 |
| 下游 | 步骤⑥ 交互链路设计 | 高风险假设的降级策略影响链路设计 |
| 下游 | 步骤⑨ 推荐方案 | 假设回检确认推荐方案的稳健性 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止只列假设不排优先级 | 资源有限，必须聚焦高风险 |
| 禁止跳过验证直接定方案 | 高风险假设未验证的方案不能进入最终推荐 |
| 禁止忽略已推翻假设 | 假设被推翻必须修正方案 |

## 6. Six Thinking Hats (de Bono)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 通过六种思维角度系统化评审方案的结构化思考方法，确保评审覆盖事实、情感、风险、价值、创新和全局 |
| 所属阶段 | 方案评审阶段（步骤⑦⑨） |
| 与10步骤关系 | **融入步骤⑦（三套方案输出）和步骤⑨（推荐方案建议）**，作为方案对比和最终推荐前的系统化评审方法 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 步骤⑦需要从三方案中做对比评审 | ✅ 必须使用 |
| 步骤⑨需要做最终推荐决策 | ✅ 必须使用 |
| 多方案难以取舍、需要结构化比较 | ✅ 必须使用 |
| 简单需求、方案差异明显 | ⚠️ 可简化为直接推荐 |

**使用前置条件**：步骤⑦三套方案已完成，步骤⑧异常流已补全。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 三套完整方案 | 步骤⑦产出 |
| 异常流验证结果 | 步骤⑧产出 |
| 假设清单与验证状态 | 步骤④产出（步骤⑨时回检） |
| 设计机会清单 | 步骤⑤产出 |

### 操作步骤

1. **白帽（事实）**：三方案各有什么客观数据和事实支撑？
2. **红帽（直觉）**：对每个方案的直觉感受是什么？用户会怎么感受？
3. **黑帽（风险）**：每个方案最大的风险和隐患是什么？
4. **黄帽（价值）**：每个方案的核心价值和收益是什么？
5. **绿帽（创新）**：是否有融合各方案优点的创造性可能？
6. **蓝帽（总结）**：综合以上分析，做出推荐决策

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 六段分析 | 表格+文本 | 每顶帽的分析要点 |
| 蓝帽总结 | 文本 | 推荐方案、推荐理由、融合建议 |

**质量标准**：
- 六顶帽子必须全部覆盖，不可跳过
- 每顶帽子对三个方案都要评价
- 蓝帽总结必须有明确推荐

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 步骤⑦ 三套方案 | 待评审的方案 |
| 上游 | 步骤④ 假设清单 | 假设验证状态（步骤⑨回检时使用） |
| 下游 | 步骤⑨ 推荐方案建议 | 评审结论直接输出为推荐方案依据 |
| 下游 | 步骤⑩ 设计优先级与分期 | 推荐方案进入分期规划 |
| 下游 | User Story Mapping | 推荐方案进入版本规划 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止跳过某顶帽子 | 六顶帽子必须完整执行，否则评审有盲区 |
| 禁止在同一段落混合帽子 | 每顶帽子单独分析，保持思维纯度 |
| 禁止蓝帽没有明确推荐 | 最终必须有清晰结论 |

## 7. User Interview Five Questions (GV)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | Google Ventures的5问访谈法，通过破冰→上下文→深挖→概念反馈→优先级五阶段获取一手用户洞察 |
| 所属阶段 | 需求发现阶段（步骤①前/中） |
| 与10步骤关系 | **引用在步骤①作为输入来源**，当需求来源不够扎实时按需调用 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 需求来源模糊、缺少用户依据 | ✅ 必须使用 |
| 需要验证5 Whys推导的根因 | ✅ 推荐使用 |
| 已有充足用户研究数据 | ❌ 不需要 |
| 时间极紧无法安排访谈 | ⚠️ 用现有数据替代，标注数据局限 |

**使用前置条件**：已明确需要补充哪些用户洞察（有明确的访谈目标）。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 访谈目标/待验证假设 | 步骤①初步分析 |
| 目标用户画像 | 产品方/已有用户数据 |
| 产品/场景背景资料 | 外部资料 |

### 操作步骤

1. **破冰**（1题）：暖场问题，建立信任，进入状态
2. **上下文**（1题）：行为回溯，了解真实使用场景和习惯
3. **深挖**（1题）：动机/痛点挖掘，追问情感和决策原因
4. **概念反馈**（1题）：方案试探，测试设计方向可行性
5. **优先级**（1题）：排序选择，确认需求优先级

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 访谈提纲 | 表格 | 5个问题 + 追问提示 + 每阶段目的 |
| 访后总结 | 文本 | 关键洞察、意外发现、待验证假设 |

**质量标准**：
- 问题按递进逻辑排列
- 每个问题有至少1个追问方向
- 访后总结必须产出可操作洞察

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 产品方/用户池 | 访谈对象和背景 |
| 下游 | 5 Whys（步骤①）| 访谈洞察作为根因分析的佐证 |
| 下游 | HMW（步骤②）| 访谈洞察转化为HMW问题 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止引导性提问 | ❌ "你是不是觉得XX很难用？" → ✅ "你使用XX时感受如何？" |
| 禁止跳过破冰直接深挖 | 用户未进入状态时深挖无效 |
| 禁止只记结论不记原话 | 用户原话是洞察的重要素材 |

## 8. User Journey Map

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 以时间线为轴，映射用户在各阶段的行为、触点、情绪和机会点的可视化工具 |
| 所属阶段 | 方案设计阶段（步骤③） |
| 与10步骤关系 | **融入步骤③**，作为交互链路设计的输出格式和分析框架 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 步骤③设计多步骤交互链路 | ✅ 必须使用 |
| 涉及多触点、多页面的体验设计 | ✅ 必须使用 |
| 单页面简单操作 | ❌ 不需要 |

**使用前置条件**：步骤②已定义设计目标，用户画像已明确。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 目标用户画像 | 步骤①②产出 |
| 设计目标 | 步骤②产出 |
| 现有流程/触点信息 | 产品资料/竞品分析 |

### 操作步骤

1. **定义人物**：明确旅程主体（哪个用户角色的旅程）
2. **划分阶段**：将体验拆分为3-7个阶段（如：发现→了解→使用→完成→离开）
3. **映射触点**：每阶段列出用户触点、行为、思考
4. **记录情绪**：标注每阶段用户情绪曲线（正面/中性/负面）
5. **识别机会**：在情绪低谷和触点断裂处标注设计机会

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 旅程地图 | 横向时间线表格 | 阶段、用户行为、触点、情绪曲线、痛点、设计机会 |

**表格结构**：

| 阶段 | 用户行为 | 触点 | 情绪 | 痛点 | 设计机会 |
|------|----------|------|------|------|----------|

**质量标准**：
- 阶段划分有逻辑，不遗漏关键环节
- 情绪标注有依据（来自用户研究或合理推断）
- 设计机会必须与痛点直接对应

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | HMW（步骤②）| HMW问题指引旅程分析重点 |
| 上游 | Competitive Analysis | 竞品旅程作为参照 |
| 下游 | 步骤③ 交互链路 | 旅程地图直接转化为交互链路 |
| 下游 | 步骤④ 三方案设计 | 旅程中的机会点驱动方案设计 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止只画主路径忽略分支 | 必须覆盖关键分支路径 |
| 禁止不标情绪曲线 | 情绪曲线是识别机会的核心工具 |
| 禁止机会点不关联痛点 | 每个设计机会必须对应明确的痛点 |

## 9. User Story Mapping (Jeff Patton)

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 以用户活动为横轴、任务细节为纵轴的二维地图，用于规划产品版本和发布节奏 |
| 所属阶段 | 版本规划阶段（步骤⑦之后） |
| 与10步骤关系 | **步骤⑦之后**，将最终方案转化为可排期的版本计划 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 最终方案涉及多版本交付 | ✅ 必须使用 |
| 需要与产品/开发对齐排期优先级 | ✅ 必须使用 |
| 单次交付的简单需求 | ❌ 不需要 |

**使用前置条件**：步骤⑦已输出最终推荐方案。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 最终推荐方案 | 步骤⑦产出 |
| 完整交互链路 | 步骤③⑤产出 |
| 业务优先级约束 | 产品方输入 |

### 操作步骤

1. **定义旅程**：以用户视角描述完整使用旅程（横轴）
2. **构建骨架**：将旅程拆分为核心活动，排列在顶部
3. **添加任务**：每个活动下拆分具体用户任务（纵轴向下展开）
4. **添加故事**：每个任务下补充具体用户故事（"作为XX，我想做YY，以便ZZ"）
5. **画发布切片**：用横线切出MVP/V1/V2等版本范围

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 故事地图 | 二维矩阵图/表格 | 活动→任务→故事，版本切片标记 |

**质量标准**：
- 横轴完整覆盖用户旅程
- 纵轴任务按优先级从上到下排列
- 每个版本切片可独立交付价值

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 步骤⑦ 最终方案 | 推荐方案的完整功能范围 |
| 上游 | User Journey Map | 旅程阶段作为横轴骨架 |
| 下游 | 开发排期 | 版本切片直接对接开发Sprint计划 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止MVP切片不包含完整核心旅程 | MVP必须能走通主流程 |
| 禁止按技术模块切版本而非用户价值 | 切片必须按用户可感知的价值划分 |
| 禁止忽略纵轴优先级 | 高优先级任务必须在上方，先进入版本 |

## 10. Competitive Analysis Framework

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 系统化收集、对比、分析竞品体验的设计研究框架，为方案决策提供外部参照 |
| 所属阶段 | 方案参考阶段（步骤⑦） |
| 与10步骤关系 | **引用在步骤⑦**，为三套方案输出提供竞品参照和设计灵感 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 需求涉及竞品已有功能 | ✅ 必须使用 |
| 需要对标行业最佳实践 | ✅ 推荐使用 |
| 全新品类、无竞品可参考 | ❌ 不需要 |
| 时间极紧的内部优化 | ⚠️ 可简化为1-2个竞品快速对比 |

**使用前置条件**：已识别2-5个主要竞品。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 竞品列表 | 产品方提供 / 自行搜索 |
| 竞品产品访问权限 | 截图/录屏/实际体验 |
| 分析维度 | 步骤②设计目标 + 步骤③链路关键节点 |

### 操作步骤

1. **识别竞品**：确定2-5个直接/间接竞品
2. **收集数据**：按分析维度截图、录屏、记录体验
3. **建对比矩阵**：功能×竞品的二维表格，标注有无/优劣
4. **分析模式**：识别共性做法、差异化策略、空白机会
5. **推导建议**：转化为对本方案的设计建议（学习/规避/超越）

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 竞品对比矩阵 | 表格 | 功能维度×竞品，评级（优/良/无） |
| 定位地图 | 图/描述 | 竞品在关键维度上的相对位置 |
| 差距分析 | 表格 | 我方vs竞品的差距项和机会项 |
| 战略建议 | 列表 | 学习/规避/超越的具体建议 |

**质量标准**：
- 至少覆盖2个竞品
- 对比维度必须与本方案设计目标相关
- 建议必须可操作，不能只是"参考竞品"

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 步骤② 设计目标 | 确定分析维度 |
| 上游 | 步骤③ 链路设计 | 确定重点对比的链路节点 |
| 下游 | 步骤⑤ 三方案输出 | 竞品参照影响方案设计选择 |
| 下游 | User Journey Map | 竞品旅程作为对照参照 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止只截图不分析 | 截图是素材不是结论，必须有分析推导 |
| 禁止照搬竞品方案 | 竞品做法需结合自身场景判断适用性 |
| 禁止忽略竞品失败案例 | 竞品放弃的功能同样有参考价值 |

## 11. Problem-Solution Fit

### 定位

| 维度 | 说明 |
|------|------|
| 一句话定义 | 验证方案是否真正解决目标问题的匹配度检查关卡，确保方案不是"自嗨"而是"对症" |
| 所属阶段 | 验证关卡（步骤⑥⑦之间） |
| 与10步骤关系 | **步骤⑥⑦之间的验证关卡**，异常流补全后、最终方案推荐前的Go/No-Go检查 |

### 适用场景

| 场景 | 是否使用 |
|------|----------|
| 方案复杂度高、涉及多环节改动 | ✅ 必须使用 |
| 需求来源方对方案有疑虑 | ✅ 必须使用 |
| 方案基于充分验证、风险低 | ⚠️ 可简化为快速检查清单 |

**使用前置条件**：步骤⑤三方案已输出，步骤⑥异常流已补全。

### 输入要求

| 输入项 | 来源 |
|--------|------|
| 原始设计问题 | 步骤②定义的核心问题 |
| 三套方案 | 步骤⑤产出 |
| 假设验证结果 | Assumption Validation Board |
| 用户反馈数据 | User Interview 5Q / 测试 |

### 操作步骤

1. **问题验证**：重新审视步骤②定义的核心问题是否仍然成立
2. **记录替代方案**：用户当前用什么方式解决此问题（替代方案/绕行方案）
3. **展示demo**：将推荐方案的核心链路呈现给用户/利益相关方
4. **评估匹配度**：方案是否直接解决核心问题？解决程度如何？
5. **迭代或推进**：匹配度≥70%则推进到步骤⑦；<70%则回退修正

### 输出规范

| 产出 | 格式 | 必填字段 |
|------|------|----------|
| 匹配报告 | 表格+文本 | 核心问题、方案对应关系、匹配度评分(%)、替代方案对比、Go/No-Go决策 |

**质量标准**：
- 必须逐条对照核心问题检验方案
- 匹配度低于70%必须回退
- 必须记录"为什么不选其他方案"

### 与其他框架的衔接

| 方向 | 框架 | 衔接内容 |
|------|------|----------|
| 上游 | 步骤⑥ 异常流补全 | 异常场景验证结果 |
| 上游 | Assumption Validation Board | 假设验证结论 |
| 下游 | 步骤⑦ 最终方案定义 | 通过关卡后进入最终推荐 |
| 下游 | Six Thinking Hats | 通过关卡的方案进入六帽评审 |

### 约束与禁忌

| 禁忌 | 说明 |
|------|------|
| 禁止跳过此关卡直接推荐方案 | 高风险方案必须经过P-S Fit检查 |
| 禁止自说自话式评估 | 必须有用户或利益相关方的真实反馈 |
| 禁止匹配度不达标仍推进 | <70%必须回退，不允许"差不多就行" |

## 第三部分：知识库引用规范

### 1. 什么时候该查知识库

| 任务类型 | 是否必须先查知识库 | 说明 |
|----------|-------------------|------|
| 产品功能/流程/规则类问题 | ✅ 必须 | 如"某功能怎么用""审批流程是什么""支持哪些格式" |
| 设计规范/组件用法查询 | ✅ 必须 | 如"按钮有几种状态""间距标准是多少" |
| 竞品信息/行业数据查询 | ✅ 必须 | 如"竞品A的上次更新做了什么""行业平均转化率" |
| 用户画像/调研数据查询 | ✅ 必须 | 如"目标用户的核心痛点是什么""NPS得分趋势" |
| 创意发散/灵感探索 | ❌ 不需要 | 创意发散阶段需要开放性思维，知识库可能限制想象 |
| 通用UX原则/方法论问题 | ❌ 不需要 | 通用设计原则（如格式塔、菲茨定律）不依赖特定知识库 |
| 用户明确说"我觉得/我认为" | ❌ 不需要 | 用户表达主观偏好时，不应以知识库反驳 |

### 2. 查询方法

#### 关键词构造策略

| 策略 | 说明 | 示例 |
|------|------|------|
| **核心实体+动作** | 产品名+功能动词 | "创客贴 导出视频" |
| **问题症状描述** | 用户遇到的具体问题 | "上传失败 报错提示" |
| **场景关键词** | 使用场景而非技术术语 | "新手引导 第一次使用" |
| **对比查询** | 需要差异信息时 | "基础版 vs 专业版 功能区别" |

#### 结果筛选规则

| 规则 | 操作 |
|------|------|
| **时效性优先** | 优先采信最新版本的信息，忽略过期条目 |
| **来源可信度** | 官方文档 > 内部Wiki > 用户反馈 > 第三方分析 |
| **交叉验证** | 关键信息至少2个来源交叉确认 |
| **结果不足时** | 变换关键词重试1-2次，仍无结果则标注"知识库未覆盖" |

### 3. 引用格式

#### 标准引用格式

```
[知识库引用] 内容摘要
- 来源：[知识库名称/文档名称]
- 层级：[L1/L2/L3/L4]
- 置信度：[高/中/低]
- 更新时间：[如可获取]
```

#### 置信度判定标准

| 置信度 | 判定条件 |
|--------|----------|
| **高** | 来自官方文档、最近3个月内更新、多来源一致 |
| **中** | 来自内部Wiki、6个月内更新、单一来源但逻辑自洽 |
| **低** | 超过6个月未更新、单一来源且无法交叉验证、来自用户反馈 |

### 4. 优先级规则

当不同信息来源发生冲突时，按以下优先级处理：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| **P0 最高** | 用户明确声明的信息 | 用户说"我们的规则是XX"，以此为准 |
| **P1** | L1产品事实层知识库 | 产品功能、技术规则等客观事实 |
| **P2** | L2设计规则层知识库 | 设计规范、交互规则 |
| **P3** | 联网搜索结果 | 用于补充知识库未覆盖的外部信息 |
| **P4** | L3竞品情报层知识库 | 竞品分析数据 |
| **P5** | L4用户知识层知识库 | 用户画像、调研数据 |

**冲突处理流程**：

1. 用户明确声明 → 直接采用，标注"基于用户输入"
2. 知识库间冲突 → 高优先级层覆盖低优先级层，标注冲突来源
3. 知识库与搜索冲突 → 优先采信知识库（更贴近产品实际），但标注搜索差异
4. 无法判断时 → 列出冲突信息，交由用户决策

### 5. 禁止行为

| 禁止行为 | 说明 |
|----------|------|
| **禁止编造知识库条目** | 查不到就说查不到，不能虚构来源 |
| **禁止引用过期信息不标注** | 超过6个月的信息必须标注时效性风险 |
| **禁止用知识库反驳用户主观偏好** | 知识库用于事实层面，不用于否定用户审美/偏好 |
| **禁止在无置信度标注的情况下引用** | 每条引用必须标注置信度 |
| **禁止将低置信度信息作为决策依据** | 低置信度只能作为参考，不能作为方案推荐的主要依据 |
| **禁止跨层混用** | L1产品事实不能和L3竞品情报混为一谈，引用时必须标明层级 |
| **禁止忽略知识库空白** | 当知识库未覆盖某领域时，必须明确告知而非用其他层信息替代 |

### 6. 知识库层级与引用场景

| 层级 | 名称 | 包含内容 | 典型引用场景 |
|------|------|----------|-------------|
| **L1** | 产品事实层 | 产品功能列表、技术规则、API能力、支持格式、性能指标、版本差异 | 步骤①需求转译时确认产品能力边界；步骤③链路设计时确认技术可行性 |
| **L2** | 设计规则层 | 设计规范、组件库用法、交互模式标准、品牌指南、可访问性标准 | 步骤③链路设计时引用标准交互模式；步骤④方案设计时引用组件规范 |
| **L3** | 竞品情报层 | 竞品功能对比、竞品更新记录、行业基准数据、市场份额 | 步骤⑤方案输出时引用Competitive Analysis数据；对标分析 |
| **L4** | 用户知识层 | 用户画像、用研报告、NPS/满意度数据、用户行为数据、反馈摘要 | 步骤①需求转译时引用用户痛点；步骤②目标定义时引用用户数据 |

#### 各步骤推荐查阅的知识库层级

| Skill步骤 | 推荐查阅层级 | 目的 |
|-----------|-------------|------|
| ① 需求转译 | L4（用户知识）+ L1（产品事实） | 确认用户真实痛点 + 产品能力边界 |
| ② 目标定义 | L4（用户知识）| 用数据支撑设计目标设定 |
| ③ 链路设计 | L1（产品事实）+ L2（设计规则）| 确保链路符合技术约束和设计规范 |
| ④ 方案设计 | L2（设计规则）| 引用标准组件和交互模式 |
| ⑤ 方案输出 | L3（竞品情报）+ L2（设计规则）| 竞品参照 + 规范校验 |
| ⑥ 异常补全 | L1（产品事实）| 确认系统实际支持的异常处理能力 |
| ⑦ 最终方案 | L4（用户知识）+ L1（产品事实）| 最终验证方案与用户需求和产品能力的匹配度 |
