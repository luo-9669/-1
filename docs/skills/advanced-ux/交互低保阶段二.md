# UX 页面交互文档生成规范

> 本文档定义从产品需求到完整页面交互文档的产出规范，覆盖所有页面的识别、设计、线框图与方案输出。
>
> **配套文档**：
> - 《高级UX思考总Skill.md》— 定义完整13节工作流（对应方法论10步骤）的目标、方法论与判断逻辑
> - 《UX方法论框架使用规范.md》— 定义每个步骤的格式模板、编号规范与自检清单
>
> 本文档聚焦于**步骤⑥之后的页面级产出**（对应总Skill的第6节"整体交互链路"及之后的方案设计），是所有页面交互文档的直接执行标准。

---

## 一、定位与适用范围

| 维度 | 说明 |
|------|------|
| **定位** | 从需求分析结果出发，产出所有页面的完整交互框架与说明文档 |
| **上游依赖** | 高级UX思考总Skill的步骤①~⑤已完成的产出（需求转译、目标定义、用户场景、假设验证、设计机会） |
| **下游衔接** | 步骤⑥~⑩的链路设计、三套方案、异常流补全、推荐方案、设计优先级与分期 |
| **产出物** | 页面交互文档（.md）+ 低保真线框图 + 状态机 + 流程图 + 3套页面方案 |

### 什么时候需要产出本文档定义的内容

| 场景 | 是否需要 | 产出范围 |
|------|---------|---------|
| 新产品/新功能的完整交互设计 | ✅ 必须 | 全文档所有模板 |
| 已有产品的局部优化（≤3个页面改动） | ⚠️ 可选，按需产出改动页面 | 仅逐页交互说明+线框图 |
| 纯视觉改版（交互不变） | ❌ 不需要 | - |
| 运营活动页（无复杂交互） | ❌ 不需要 | - |
| 新增弹窗/浮层（不改动页面） | ⚠️ 可选 | 弹窗定义表+交互规则 |
| 新增状态变化（已有页面） | ⚠️ 可选 | 状态机+异常状态表 |

> 💡 **产出范围判断原则**：如果改动涉及新的页面跳转或新的数据流，就需要完整的流程+状态+逐页文档。如果只是已有页面内部的状态变化或弹窗新增，可以只产出相关章节。判断标准：**是否有新的页面间跳转**。

## 一-B、与高级UX思考总Skill的衔接（阶段一 → 阶段二）

### 衔接的本质

阶段一不是一个"跑完10步就结束"的流程——它是**多轮对话迭代**的过程。用户在对话中会：
- 不断补充和修正需求细节
- 上传参考文档、竞品截图、业务规则等附件
- 与 Agent 讨论确认边界条件和约束
- 在多个方案间权衡后做出选择

最终方案是**对话过程中逐步收敛**出来的，不是机械执行10步的结果。

### 阶段二需要吸收的输入来源

进入阶段二时，需要从以下**三类来源**综合提取信息：

| 来源 | 包含什么 | 为什么重要 |
|------|---------|-----------|
| **阶段一最终产出文档** | 7步骤的完整结构化输出（需求清单、目标矩阵、用户画像、页面三件套、方案对比、决策卡片等） | 这是显性的、结构化的基础数据 |
| **对话过程中的变更和补充** | 用户在多轮对话中补充的细节描述、确定的边界条件、上传的参考文档、明确的约束要求、方案调整决策 | 这些内容可能**没有体现在最终产出文档中**，但直接影响页面设计 |
| **Agent 上下文中的隐性信息** | Agent 在对话中做出的判断、与用户确认过的假设、讨论过的取舍理由、识别出的风险点 | 这些是设计决策的依据，需要在阶段二保持一致 |

### 衔接操作步骤

用户手动进入阶段二时，应提供以下材料：

**必须提供：**
- 阶段一的最终产出文档（完整10步骤结果）
- 对话过程中产出的补充文档（如有）

**建议提供：**
- 对话过程中上传的关键附件（参考文档、竞品截图等）
- 明确的变更说明：哪些内容在对话中被修改/补充/确认了（如有）

**如果用户只提供最终产出文档：**
- Agent 应主动询问：对话过程中是否有重要变更、补充决策或约束条件未体现在文档中？
- 避免遗漏对话中确认过但未写入文档的关键信息

### 产出覆盖要求

- 最终方案页面清单中的**每个页面**都必须生成完整的交互文档（信息架构表 + 主流程表 + 状态机 + 交互规则表 + 异常状态表 + 低保真线框图 + 弹窗定义）
- N 个页面 = N 份完整页面交互文档 + 1 份全局交互规范汇总
- 不允许遗漏页面清单中的任何页面
- 页面交互文档中的设计决策必须与阶段一对话中确认的方案一致，不能与对话结论矛盾

## 二、核心流程

```
需求快析 → 页面识别与编号 → 信息架构实体表 → 主流程与状态机 → 弹窗定义 → 逐页交互说明 → 关键断点与优化节点 → 核心页面线框图 → 全局交互规范 → 页面3方案 → 方案验证与收敛 → 文档组装
```

### 流程各阶段依赖关系

| 阶段 | 输入 | 产出 | 依赖前置 |
|------|------|------|---------|
| 需求快析 | 上游步骤①②③产出 | 核心假设清单 | 无 |
| 页面识别与编号 | 核心假设 + 核心场景 | 页面总览表 | 需求快析 |
| 信息架构实体表 | 核心假设 + 核心对象 | 实体/属性/关系/状态表 | 需求快析 |
| 主流程与状态机 | 页面总览表 + 实体表 | 步骤表 + 状态机 + 页面流转图 | 页面识别 |
| 弹窗定义 | 主流程中的交互需求 | 弹窗与抽屉定义表 | 主流程 |
| 逐页交互说明 | 上述所有产出 | 每页的框架/规则/异常/线框图 | 弹窗定义 |
| 关键断点 | 主流程 + 逐页说明 | 断点分析表 + 优化建议 | 逐页交互 |
| 全局交互规范 | 所有页面共性规则 | 7类规范表 | 逐页交互 |
| 页面3方案 | 核心页面 + 断点分析 | 3方案 + 对比矩阵 + 选择建议 | 全局规范 |
| 方案验证与收敛 | 3方案产出 | P-S Fit + 决策卡片 + 埋点 + 待确认 | 页面3方案 |
| 文档组装 | 所有上述产出 | 最终交付文档 | 全部 |

## 三、统一编号体系

> **所有编号必须全局唯一、跨文档一致**，禁止同一编号指向不同对象。

### 编号规则总表

| 对象类型 | 编号格式 | 起始值 | 示例 | 说明 |
|----------|---------|--------|------|------|
| 页面 | P + 两位数字 | P01 | P01, P02, P15 | 仅用于完整页面，不用于弹窗 |
| 状态机状态 | ST + 数字 | ST0 | ST0, ST1, ST5 | ST0 = 初始/空态 |
| 主流程步骤 | S + 数字 | S1 | S1, S2, S10 | 按执行顺序递增 |
| 弹窗/抽屉 | M + 两位数字 | M01 | M01, M02, M03 | 用于弹窗、抽屉、浮层 |
| 交互规则 | IR + 数字 | IR1 | IR1, IR2, IR5 | 按页面内顺序递增 |
| 异常状态 | E + 数字 | E1 | E1, E2, E4 | 按页面内顺序递增 |

### 编号引用可读性规则

> 💡 **填写要求**：编号定义列可以只写编号；凡是“来源 / 依据 / 关联 / 承接 / 去向 / 页面 / 状态 / 弹窗 / 规则 / 异常”这类引用字段，禁止裸编号引用，必须写成 `编号 - 名称/说明`。例如不要只写 `P01`、`M03`、`ST2`、`IR5`，要写成 `P01 - 爆款复刻首页`、`M03 - 生成排队弹窗`、`ST2 - 生成中状态`、`IR5 - 点击生成视频`。

### 编号联动规则

> 💡 **填写要求**：编号联动是文档一致性的关键。所有交叉引用必须双向可追溯：交互规则中引用的弹窗/状态机，必须能在弹窗定义表和状态机表中找到对应条目；线框图中标注的 IR 编号，必须能在交互规则表中找到对应规则。建议在文档完成后做一次"编号联动扫描"，逐一核对。

| 联动关系 | 说明 | 示例 |
|----------|------|------|
| 交互规则 ↔ 弹窗编号 | 交互规则中触发弹窗时，必须引用弹窗 M 编号并带名称 | "点击删除 → 弹出 M03 - 删除确认弹窗" |
| 交互规则 ↔ 状态机 | 交互规则触发状态变化时，必须引用 ST 编号并带名称 | "保存成功 → 状态迁移至 ST4 - 保存成功" |
| 线框图注释 ↔ 弹窗/状态机 | 线框图中标注的交互区域必须引用 M 或 ST 编号并带名称 | "① 点击区域 → 触发 M02 - 上传抽屉 / 进入 ST3 - 上传中" |
| 主流程步骤 ↔ 页面编号 | 每个步骤必须指向对应页面并带页面名 | "S3 - 确认订单 → P04 - 订单确认页" |

**编号联动示例（以AI视频爆款复刻为例）**：

以下展示一组完整的编号联动链条：

```
交互规则 IR1（PG01首页）：
  用户粘贴链接 → 点击"开始解析"
  → 触发弹窗 M01 - 解析进度弹窗         ← 弹窗定义表中可查
  → 状态迁移 ST0 - 初始 → ST1 - 解析中  ← 状态机表中可查

交互规则 IR6（PG05生成预览页）：
  用户点击"重新生成"
  → 触发弹窗 M03 - 生成排队弹窗          ← 弹窗定义表中可查
  → 状态迁移 ST2 - 预览 → ST3 - 排队中  ← 状态机表中可查

主流程步骤 S6：
  用户点击"生成" → P05 - 生成预览页 → 触发 M03 - 生成排队弹窗
  → 对应状态机 ST2（生成预览页状态机）

线框图标注（PG05）：
  ① 视频预览区 → IR1-IR4 交互区域
  ② 脚本对照面板 → IR5 交互区域
  ③ 参数调节面板 → IR6 交互区域
  ④ 底部操作栏 → IR7-IR9，引用 M03 - 生成排队弹窗 / M04 - 导出设置弹窗
```

### 页面编号范围约定

| 编号范围 | 模块分类 | 说明 |
|----------|---------|------|
| P01-P09 | 核心交易流程 | 用户完成核心任务必须经过的页面 |
| P10-P19 | 账户与设置 | 个人中心、订单列表、设置等 |
| P20-P29 | 辅助页面 | 搜索结果、帮助、关于、协议等 |
| P30-P39 | 异常/特殊页面 | 错误页、维护页、空态引导页等 |

## 四、需求快析

> 承接总Skill步骤①②③的产出，此处只做轻量提取，不做完整分析。

### 必须提取的信息

> 💡 **填写要求**：需求快析的信息提取是后续所有步骤的基础。6 个维度必须逐一确认，不可跳过。如果上游步骤①②③的产出中已有明确信息，直接引用并标注置信度"高"；如果是推断得出，标注"中"或"低"。缺失维度必须标注"缺失"并给出降级处理方式（推断/追问/默认值）。

| 维度 | 提取内容 | 如果缺失 |
|------|---------|---------|
| 产品类型 | App / 小程序 / Web / H5 | 询问用户 |
| 核心场景 | 用户用这个产品做什么（1-3个核心场景） | 基于产品类型推断，标注置信度低 |
| 目标用户 | C端/B端/内部，用户画像 | 基于产品类型推断，标注置信度低 |
| 履约/交付方式 | 线上/线下/混合，如何交付价值 | 基于产品类型推断 |
| 核心对象 | 主要数据实体（如订单、商品、用户） | 从场景推断 |
| 已有信息 | 用户是否提供了功能列表、流程图、竞品参考 | 按已有信息推进 |

**提取示例（以AI视频爆款复刻为例）**：
```
| 维度 | 提取内容 | 置信度 | 来源 |
|------|---------|--------|------|
| 产品类型 | Web工具 | 高 | 需求原文明确 |
| 核心场景 | 追热点复刻带货视频/对标学习复刻/批量素材生产 | 中 | 蝉镜官网+竞品分析 |
| 目标用户 | 电商运营/知识博主/MCN代运营 | 中 | 蝉镜用户画像 |
| 履约/交付方式 | 线上：AI生成视频→下载/发布 | 高 | 需求原文明确 |
| 核心对象 | 爆款视频/拆解结果/脚本/素材/生成任务/复刻视频 | 高 | 竞品功能拆解 |
| 已有信息 | 竞品参考（蝉镜/凡科/Kickart/可灵）、用户画像描述 | 高 | 步骤①②产出 |
```

### 产出格式

> 💡 **填写要求**：核心假设至少覆盖 5 个维度：产品类型、核心场景、目标用户、MVP范围、核心对象。每条假设必须标注置信度（高/中/低）。高=用户明确说明或有文档依据；中=基于竞品或行业惯例推断；低=基于经验直觉。置信度为"低"的假设必须在后续步骤中标注并尽量通过追问或竞品调研确认。如果某个维度完全无信息，标注"缺失"并给出默认推断。

```markdown
## 核心假设

1. 产品类型：Web工具（置信度：高，用户明确说明）
2. 核心场景：发现爆款视频→AI拆解结构→替换内容→一键生成同款（置信度：中，基于竞品蝉镜/凡科/Kickart推断）
3. 目标用户：电商运营/知识博主/MCN代运营（置信度：中，基于蝉镜用户画像推断）
4. MVP 范围：单条复刻完整流程（导入→拆解→定制→替换→生成→导出），不含批量生成（置信度：中，基于行业惯例推断）
5. 核心对象：爆款视频、拆解结果、定制脚本、素材槽位、生成任务、复刻视频（置信度：高，基于竞品功能拆解）
```

**填写示例（以AI视频爆款复刻为例）**：
```
## 核心假设

1. 产品类型：Web工具（置信度：高，用户明确说明）
2. 核心场景：电商运营追热点复刻带货视频（置信度：高，蝉镜核心场景）

5. 核心对象：爆款视频/拆解结果/分镜/定制脚本/素材槽位/数字人/生成任务/复刻视频/模板/用户项目（置信度：高）
```

## 五、页面识别与编号

### 识别检查清单

> 💡 **填写要求**：页面识别是文档的基础步骤，必须按以下 6 项检查清单逐一确认，不可跳过。每项检查的结果直接体现在页面总览表中。如果发现遗漏，必须补充页面后再进入下一步。

- [ ] 用户从打开到完成核心任务，经过哪些页面？（必须覆盖主流程所有节点）
- [ ] 每个页面中是否有弹窗/抽屉/浮层？（弹窗/抽屉/浮层分配 M 编号，不占用页面 P 编号）
- [ ] 异常路径是否需要独立页面？（如错误页、空态页、权限页）
- [ ] 辅助功能是否需要页面？（如帮助、设置、关于）
- [ ] 不同角色是否看到不同页面？（如管理员 vs 普通用户）
- [ ] 页面间数据传递是否需要中间状态页面？（如加载页、确认页）

**页面识别示例（以AI视频爆款复刻为例）**：

按检查清单逐项确认：
1. **主流程页面**：导入→拆解→定制→替换→生成预览→结果 = 6个页面（P01-P06）
2. **弹窗/浮层**：解析进度(M01)、素材上传(M02)、生成排队(M03)、导出设置(M04)、合规提醒(M05)、新手引导(M06) = 6个弹窗
3. **异常页面**：无独立异常页（异常通过弹窗和页面内状态处理）
4. **辅助页面**：模板广场(P07)、自由创作(P08) = 2个页面
5. **角色区分**：一期不区分角色（所有用户相同流程）
6. **中间状态页**：无（状态通过弹窗承载）

→ 最终识别出 **8个页面 + 6个弹窗 = 14个独立界面单元**

### 产出：页面总览表

| 编号 | 页面名称 | 类型 | 所属模块 | 核心职责 | 对应步骤 |
|------|---------|------|---------|---------|---------|

### 7.2 状态机

#### 判定是否需要状态机

> 💡 状态机是描述页面/组件在不同条件下如何变化的一组规则。判断标准：如果页面的视觉表现或可用操作会因条件不同而改变，就需要状态机。

| 条件 | 是否需要 | 说明 |
|------|---------|------|
| 有"订单/任务/工单"概念 | ✅ 必须有 | 生命周期状态复杂 |
| 有"审核/审批"流程 | ✅ 必须有 | 多角色、多状态流转 |
| 页面有 ≥2 种可区分状态 | ✅ 必须有 | 如加载中/有数据/空态 |
| 纯静态展示页面 | ❌ 不需要 | 无状态变化 |
| 仅有 loading/loaded 两种状态 | ⚠️ 可选 | 简单状态可合并到异常状态表 |

#### 格式选择规则

| 条件 | 格式 |
|------|------|
| 状态数 ≤5 | 状态机表格 |
| 状态数 >5 或分支复杂 | Draw.io 状态图 |

#### 状态机表格格式

**[页面/组件名称] 状态机**

状态定义：

| 状态ID | 状态名称 | 触发条件 | 页面表现 | 可执行操作 |
|--------|----------|----------|----------|------------|
| ST0 | 初始/空态 | 首次进入，无数据 | 显示空态引导 | 点击创建 |
| ST1 | 加载中 | 触发数据请求 | 骨架屏/Loading | 无 |
| ST2 | 默认展示 | 数据加载完成 | 展示内容列表 | 编辑/删除/详情 |
| ... | ... | ... | ... | ... |

状态迁移图：
```
ST0(初始) ──[触发事件]──→ ST1(状态) ──[事件]──→ ST2(状态)
                              └──[异常]──→ ST_error
```

状态迁移规则：

| 当前状态 | 触发事件 | 目标状态 | 迁移条件 | 过渡动画 | 备注 |
|----------|----------|----------|----------|----------|------|
| ST0 | 用户点击XX | ST1 | 条件成立 | fade-in | - |
| ... | ... | ... | ... | ... | ... |

> 💡 **填写要求**：每个有状态变化的页面/组件必须独立输出状态机。状态定义表中"状态ID"从 ST0 开始，ST0 固定为初始/空态。"触发条件"必须区分用户操作、系统自动、第三方回调三类触发源。"页面表现"必须写清用户看到什么具体变化（如"弹出解析进度弹窗"而非"有反馈"）。状态迁移规则表中"过渡动画"可填"fade-in/slide-left/none"等。所有状态迁移必须闭环，不能有孤立状态。通常核心页面状态数 3-8 个。

**填写示例（以AI视频爆款复刻为例 — 生成预览页状态机）**：
```
状态定义：

| 状态ID | 状态名称 | 触发条件 | 页面表现 | 可执行操作 |
|--------|----------|----------|----------|------------|
| ST2-排队中 | 等待生成 | 点击生成且资源繁忙 | M03 - 生成排队弹窗展示排队位置+预估时间 | 后台等待/取消 |

| ST2-预览 | 生成完成 | AI生成成功 | 视频播放器+脚本对照+参数面板 | 播放/重新生成/局部修改/导出 |

状态迁移图：
ST2-排队中 ──[排队完成]──→ ST2-生成中 ──[生成成功]──→ ST2-预览
                                └──[生成失败]──→ ST2-失败
ST2-预览 ──[重新生成]──→ ST2-排队中（M03 - 生成排队弹窗）
ST2-预览 ──[局部修改]──→ ST2-编辑
ST2-预览 ──[导出]──→ ST2-导出（M04 - 导出设置弹窗）
ST2-失败 ──[重试]──→ ST2-排队中（M03 - 生成排队弹窗）

#### 约束

- ST0 固定为初始/空态
- "触发事件"必须写清是谁触发的（用户操作 / 系统自动 / 第三方回调）

- 所有状态迁移必须闭环，不能有孤立状态

### 页面框架表格填写要求

> 💡 **填写要求**：每个页面必须输出页面框架表格，区域数量通常 3-6 个。"区域"从页面顶部到底部按顺序排列。"内容"列写具体包含的 UI 元素（如"Logo + 功能Tab + 用户头像"），不写"导航栏内容"这种笼统描述。"说明"列标注定位方式（固定/吸顶/吸底/随滚动/浮层）和布局方式（列表/网格/卡片/双栏）。

**填写示例（以AI视频爆款复刻 — PG03 脚本定制页为例）**：
```
| 区域 | 内容 | 说明 |
|------|------|------|
| 顶部步骤条 | ①导入 → ②拆解 → ③定制 → ④替换 → ⑤生成 | 当前步骤③高亮，固定顶部 |
| 左侧参考面板 | 原视频分镜结构（只读），含时间轴+脚本预览 | 占35%宽度，可折叠 |
| 右侧定制面板 | 业务信息表单（行业/商品/卖点）+ AI生成脚本区 | 占65%宽度，自适应滚动 |
| 底部操作栏 | [上一步] [AI生成脚本] [下一步] | 吸底固定 |
```

### 交互规则编写要求

1. **每页至少 5 条**交互规则
2. **反馈具体**：不写"有反馈"，写具体反馈内容（如"底部 Toast 提示'已加入购物车'，购物车角标+1"）
3. **联动关系**：如果页面内有联动（如左右栏联动滚动），必须说明
4. **价格/数量联动**：涉及价格计算、数量变化的，说明实时更新规则
5. **引用编号**：触发弹窗必须引用 `M 编号 - 弹窗名称`，状态变化必须引用 `ST 编号 - 状态名称`

> 💡 **填写要求**：交互规则表是逐页交互说明的核心产出。每页独立输出一张表，至少 5 条规则，核心页面通常 7-12 条。"用户操作"列写具体的用户行为（点击/滑动/长按/拖拽/输入），不写模糊描述。"系统反馈"列写用户能看到的具体变化，包含视觉反馈、Toast 内容、弹窗编号引用、状态机状态引用。"备注"列补充边界条件、触发频率限制、联动关系等。

**填写示例（以AI视频爆款复刻 — PG03 脚本定制页为例）**：
```
| 用户操作 | 系统反馈 | 备注 |
|---------|---------|------|
| 填写行业字段 | 自动联想行业标签（如"电商/美妆/教育"） | 支持下拉选择+手动输入 |
| 填写商品名称 | AI 自动提取核心卖点关键词 | 实时生成，输入停顿500ms后触发 |
| 点击"AI辅助改写" | 光标位置脚本段落闪烁重写动画 | 保留原结构，替换内容 |
| 点击分镜行展开 | 左侧面板展示该分镜详细参数（时长/场景/技巧） | 手风琴折叠，同时只展开一行 |
| 修改脚本文案 | 右侧脚本区域实时更新字数统计 | 逐字计数，上限500字 |
| 点击"AI生成脚本" | 按钮变为loading，3-8s后展示生成结果 | 基于拆解结构+业务信息 |
| 点击"换一换" | AI重新生成一版不同角度的脚本 | 最多5次，超出提示"已穷尽变体" |
```

### 异常状态编写要求

1. **必须覆盖 5 种状态**：加载中（E1）、空状态（E2）、错误态（E3）、无权限（E4）、业务异常（E5+）
2. **处理方式可执行**：不写"提示用户"，写具体处理方式（如"Toast 提示'商品已售罄'，按钮置灰不可点"）
3. **关联状态机**：异常处理后的状态迁移必须明确（如"重试后 → ST1"）
4. **视觉区分**：每种异常状态使用不同的视觉处理方式，确保用户能快速识别

> 💡 **填写要求**：每个页面必须独立输出异常状态表。必须覆盖 5 种状态：加载中、空状态、错误态、无权限、至少1种业务异常。"表现"列写具体的视觉表现（骨架屏/插画+文案/红色提示/Toast等），不写"提示用户"。"处理方式"列写用户可执行的操作和系统行为，必须可执行。业务异常因页面特性而异（如视频解析失败、素材不匹配、余额不足等）。

**填写示例（以AI视频爆款复刻 — PG03 脚本定制页为例）**：
```
| 状态 | 表现 | 处理方式 |
|------|------|---------|
| 加载中 | AI生成中：按钮loading+脚本区域骨架屏 | 3-8s后替换为生成结果 |
| 空状态 | 未填写表单："请先填写业务信息"引导文案 | 高亮第一个未填字段 |
| 错误态 | AI生成失败：Toast"生成失败，请重试"+重试按钮 | 支持手动编辑已有内容 |
| 无权限 | 未登录用户点击"AI生成"：弹窗引导登录 | 登录后恢复当前编辑内容 |
| 业务异常-必填项未填 | "下一步"按钮置灰+字段下方红色提示 | 填写后激活按钮 |
| 业务异常-生成超时 | Toast"AI忙碌中，请稍后"+重试按钮 | 30s超时阈值 |
```

## 十二、页面3方案输出规范

> 对核心页面或关键交互节点，产出 3 个不同的设计方案，供评审讨论和选择。

### 12.1 适用范围

> 💡 **填写要求**：3方案输出仅针对核心页面和存在体验矛盾的关键节点。简单页面（帮助、关于、协议等）不需要3方案，避免产出膨胀。判断标准：该页面是否是用户完成核心任务的必经之路？是否涉及设计取舍的权衡？如果两个条件都满足，则必须出3方案。通常 1-3 个核心页面需要3方案。

| 场景 | 是否需要3方案 | 理由 |
|------|-------------|------|
| 核心交易流程的关键页面 | ✅ 必须 | 影响核心转化率，需要充分讨论 |
| 存在体验矛盾的页面 | ✅ 必须 | 如效率vs控制、深度vs简洁，需要权衡 |
| 首次使用/新手引导页面 | ✅ 必须 | 影响新用户留存，方案差异大 |
| 简单辅助页面（帮助、关于等） | ❌ 不需要 | 交互简单，无需多方案对比 |
| 纯展示页面（协议、声明等） | ❌ 不需要 | 无交互，纯内容展示 |
| 弹窗/浮层 | ⚠️ 一般不需要 | 除非弹窗承载核心交互逻辑 |

### 12.2 三种方案类型

> 💡 **填写要求**：三种方案类型是固定的分析框架，必须覆盖稳妥/效率/引导三个维度。每种方案的核心思路用一句话说明其设计策略。"适用场景"说明该方案在什么条件下最优。三种方案之间必须有明显差异化：如果三个方案只是布局微调或颜色不同，则不算有效方案。方案选择时不可"和稀泥"，必须有明确的推荐倾向。

| 方案类型 | 核心思路 | 适用场景 |
|----------|---------|---------|
| **方案A：稳妥型** | 贴近用户已有认知，改动小，易落地，风险低 | 时间紧、改动范围有限、老用户习惯强 |
| **方案B：效率型** | 强调任务完成效率，减少步骤，优化主路径 | 高频任务、用户目标明确、追求转化率 |
| **方案C：引导型/增强型** | 强调引导、教育、曝光或增长，交互干预更强 | 新用户 onboarding、新功能曝光、业务希望强化某些行为 |

### 12-B.1 P-S Fit 验证表

> 💡 **填写要求**：P-S Fit（Problem-Solution Fit）验证表用于检验每个核心问题是否有对应方案解决。"核心问题"从步骤②的 HMW 问题卡片或体验矛盾表中提取。"匹配证据"说明方案的哪个设计点解决了该问题。"未匹配风险"说明方案可能没有覆盖到的问题侧面。"验证方式"说明如何验证（用户测试/数据验证/竞品对标等）。"置信度"标注匹配判断的可信程度。每个核心问题至少 1 行，通常 4-6 行。

**填写示例（以AI视频爆款复刻为例）**：
```
| 核心问题 | 对应方案 | 匹配证据 | 未匹配风险 | 验证方式 | 置信度 |
