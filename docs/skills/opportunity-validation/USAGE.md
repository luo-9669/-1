# Opportunity Validation Skill 使用说明

## 文件说明

- `SKILL.md`：Codex/Agent 读取的主 skill 文件。
- `run-template.md`：每次运行分析时可复制的输入模板。
- `INTRO.md`：给团队成员看的能力介绍。
- `USAGE.md`：具体使用方式和输出检查清单。

## 推荐使用方式

在当前项目里，可以这样发起：

```text
使用 opportunity-validation skill 对当前项目执行一次完整机会验证。
请结合项目资料、上传文档、蓝图、交互路径、Demo、竞品流程和三轮评审，
输出用户行为意图、设计方案决策矩阵、机会点、竞品参考、差异化方案、优先级和可执行开发任务。
```

如果有明确竞品，可以补充：

```text
竞品重点看 Coze、Notion AI、飞书/Lark、Copilot Studio、Figma、Linear。
请分析他们在上传资料、生成流程、状态反馈、人工确认、产物沉淀、失败恢复上的做法，
说明他们为什么这么做，我们参考什么、不照搬什么、怎么做得更好。
```

## 输入建议

尽量提供：

```json
{
  "projectProfile": {
    "productName": "",
    "positioning": "",
    "targetUsers": "",
    "primaryGoal": "",
    "currentStage": ""
  },
  "documents": ["PRD", "交互说明", "需求补充", "用户反馈"],
  "knowledgeBase": ["历史蓝图", "设计规范", "业务规则"],
  "conversationSignals": ["关于弹窗、页面、二级页面、路径、优先级、区域重叠的当前需求讨论"],
  "competitors": ["指定竞品或参考产品"],
  "currentArtifacts": ["当前蓝图", "当前 Demo", "状态机", "页面截图"]
}
```

信息不完整时也可以运行，但输出必须标注假设和证据缺口。

## 必须执行的分析顺序

1. 建立项目上下文。
2. 定义用户角色和 JTBD。
3. 分析用户行为意图：正在做什么、想先看到什么、哪些信息不需要。
4. 从当前需求对话/设计记录里提取弹窗、页面、二级页面、路径、优先级、区域重叠证据。
5. 建设计方案决策矩阵：承载形式、路径跳转、返回状态、错误反馈、功能层级、区域位置。
6. 做竞品流程反向分析。
7. 模拟当前产品用户旅程。
8. 补服务蓝图和后台状态。
9. 提取痛点证据。
10. 建机会解决方案树。
11. 判断每个方案的产品形态、路径和区域排布。
12. 用 RICE、Kano、金字塔和竞品价值评分。
13. 做 UX、产品价值、工程可执行三轮评审。
14. 输出最终迭代计划和开发任务。

## 竞品分析必须回答

每个竞品或参考产品都要尽量回答：

- 入口在哪里？
- 用户第一步做什么？
- 是否支持空输入或示例体验？
- 上传入口放在哪里？
- 生成过程如何反馈？
- 哪些步骤需要人工确认？
- 失败、权限、空结果怎么恢复？
- 产物在哪里沉淀？
- 是否支持刷新、版本对比、继续追问？
- 如何交付给设计或研发？
- 他们为什么这样设计？
- 我们参考什么？
- 我们不应该照搬什么？
- 我们怎么做得更好？

## 输出检查清单

完成后检查：

- 是否有 `competitorWorkflowAnalysis`。
- 是否有 `behaviorIntentMatrix`、`conversationEvidenceMap` 和 `designSchemeDecisionMatrix`。
- 是否有竞品流程矩阵。
- 是否解释了竞品为什么这么做。
- 是否输出了“参考什么 / 不照搬什么 / 我们怎么更好”。
- 每个 P0/P1 机会点是否有当前产品路径证据。
- 每个 P0/P1 机会点是否说明用户想看到什么、哪些信息应降级或不展示。
- 每个 P0/P1 机会点是否说明竞品证据对优先级的影响。
- 每个方案是否说明页面、二级页面、弹窗、抽屉、新页签、内联面板等形态，以及什么时候用哪种。
- 每个方案是否说明路径跳转、返回状态、保留草稿/筛选/滚动/选中项的规则。
- 每个方案是否说明错误报在哪里、报什么、用户如何恢复。
- 每个方案是否区分主要功能、同级功能、二级功能、低优先级功能，并说明位置排布。
- 每个方案是否检查了区域重叠、视觉竞争、点击目标冲突和阅读顺序。
- 每个方案是否有状态、异常、恢复、下一步。
- 三轮评审是否都产出修改项。
- 最终任务是否能直接进入开发。

## 安全边界

只允许使用公开资料、用户授权资料和可访问页面。不要声称获得竞品内部 prompt、隐藏思考链、私有知识库、非公开 Markdown 或内部配置。
