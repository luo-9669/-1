# Opportunity Validation Skill 介绍

## 这个 Skill 解决什么问题

`opportunity-validation` 用来把项目资料、产品文档、蓝图、交互路径、Demo 和竞品流程转成可执行的产品/UX 迭代方案。

它不是单纯做需求总结，而是强制回答四个问题：

1. 用户真实路径哪里卡住了？
2. 竞品为什么这样设计，哪些值得参考，哪些不能照搬？
3. 我们怎么做能更可信、更可评审、更可执行？
4. 用户当下想做什么、想先看到什么、哪些信息其实不需要出现？
5. 最终应该先做什么、以什么形态做、如何验证？

## 核心能力

- 项目基础信息梳理：人群、目标、定位、当前阶段、主要任务。
- 用户旅程模拟：覆盖空输入、上传、解析、蓝图、Demo、失败恢复、导出、返回修改。
- 竞品流程反向分析：还原入口、输入、状态、确认、失败、产物沉淀和交付路径。
- 竞品原因分析：从产品、交互、技术、商业四个角度解释为什么这么做。
- 差异化决策：明确参考什么、不照搬什么、我们怎么做得更好。
- 用户行为意图分析：判断用户正在做什么、想看到什么、哪些信息会干扰主路径。
- 信息优先级判断：把不需要、低频、过早出现的信息降级到二级页面、抽屉、折叠区、筛选器或 P3。
- 当前需求证据提取：从弹窗、页面、二级页面、路径、优先级、区域重叠等讨论中找出 UX skill 能发挥作用的点。
- 设计方案决策矩阵：明确用弹窗、页面、二级页面、新页签还是内联反馈，路径如何跳转和返回，错误报在哪里，功能如何分主次和同级，区域放在哪里。
- 机会点排序：结合路径证据、竞品证据、RICE、Kano、功能金字塔和权重评分。
- 方案形态判断：区分页面、弹窗、抽屉、内联面板、新页签、状态页等承载方式。
- 三轮评审：UX 评审、产品价值评审、工程可执行评审。
- 最终产物：机会点 Top、证据矩阵、方案形态、迭代计划、开发任务。

## 适合什么时候用

- 想验证当前产品流程是否合理。
- 想把 PRD/资料变成蓝图和可交互 Demo。
- 想知道竞品流程为什么这样做，以及自己该怎么超越。
- 想给产品、设计、研发评审一个完整依据。
- 想决定下一轮优先做什么。

## 不适合什么时候用

- 单个按钮文案修改。
- 单纯 UI 视觉微调。
- 不需要产品/UX 判断的纯代码修复。
- 试图获取竞品非公开 prompt、内部配置、隐藏思考链或私有文档。

## 重要原则

- 没有用户路径证据，就不能定义机会点。
- 没有竞品原因分析，就不能说“参考竞品”。
- 竞品证据只能增强判断，不能替代当前产品的真实路径模拟。
- 用户当下不想看、不需要看、看了会误解的信息，应降级或不放在当前路径。
- 页面、弹窗、二级页面、抽屉和侧栏的选择必须服务用户行为，不服务数据分类本身。
- 每个方案都必须说明承载形态、状态变化、失败恢复和验收方式。
- 最终输出必须能进入开发，而不是停留在建议。
