# Interaction Lofi Canvas Contract

## Goal

The `交互低保` stage is a page-level low-fidelity interaction canvas. It must show detailed page wireframes and interaction details, not only generic module skeletons.

## Data Source Priority

Use current stage canvas nodes first:

```js
totalDesignFlow.stageCanvases['interaction-lofi'].nodes
```

Each page node should keep its structured artifact:

```js
node.pageLayoutArtifact
```

When page-level interaction instructions are available, the same page node should
also keep:

```js
node.interactionSpecArtifact
```

Fallback page data under `totalDesignFlow.pages` is only allowed when the active slice has no stage canvas nodes.

This follows the shared [Stage Artifact Rendering](stage-artifact-rendering.md) contract: stage canvas artifacts are canonical, and frontend fallbacks must not replace generated backend/model artifacts.

## Must Preserve

- Canvas cards show page-level wireframe previews from `pageLayoutArtifact.asciiWireframe`.
- The ASCII wireframe comes from backend/model `pageLayoutArtifact`, not a frontend-reconstructed business layout.
- For Advanced UX `页面交互框架与说明.md`, the backend/model document must follow the page three-piece structure before importing the canvas: `页面总览清单` with completeness statistics, `页面流转总览` with a flow relation table and text flow diagram, and `页面框架展示` for each analyzed page.
- Advanced UX imported page nodes must preserve a per-page ASCII/text layout diagram as the required framework preview. Generated gray low-fidelity images are optional downstream visual artifacts for core/reporting pages and should attach back to the matching page node by `pageId`.
- Advanced UX page interaction documents for self-owned products must include an independent `交互规则表（自有产品）` that maps `pageId + regionId` to trigger, precondition, behavior, success feedback, error feedback, edge cases, and related API when known.
- Fullscreen detail also shows the selected page wireframe.
- Fullscreen detail may provide a `框架图 / 低保原型 / 交互说明` switch when page artifacts exist. `框架图` remains the default; `低保原型` is a visualized rendering of the same backend-owned `pageLayoutArtifact`, not a second low-fidelity generation result; `交互说明` appears when `interactionSpecArtifact` exists.
- Page-level `交互说明` must render backend/model `interactionSpecArtifact.interactionRows`; it must not be authored as fixed frontend business copy.
- Page-level `交互说明` rows should be implementation-grade when available: target, gesture, enable/disable/display/hide conditions, operation result, feedback, state prompt copy, navigation/result, motion notes, state coverage, and test points.
- Page-level `交互说明` must expose backend/model state prompt copy and gesture notes when present. Generated interaction artifacts should preserve common page gestures such as `下拉手势 → 触发接口刷新页面数据` and `内容区域纵向滚动 → 顶部/底部固定区不跟随滚动`.
- Page-level `交互说明` should also expose backend/model state or flow closure when present, using `interactionSpecArtifact.transitions`, `stateTransitions`, `stateMatrix`, or interaction rows as display sources. This preview is for understanding the closed loop of one page/flow and must not duplicate the global requirement-dissection state machine.
- Older persisted runs that do not yet have `interactionSpecArtifact` may normalize existing backend/model fields such as `interactionDetails`, `interactionSpec`, and `pageArchitecture` into the same row shape for display. This compatibility path must not invent new business content.
- Fullscreen detail prioritizes the selected page wireframe. Auxiliary artifact data such as interaction details, model recommendation, frontend handoff, backend handoff, and `screenContract` must remain in `pageLayoutArtifact` for downstream generation, but should not be rendered as separate frontend cards when it would distract from the framework view.
- `fullscreenNode`, `activeNode`, and Agent node lookup must use the same current-stage node pool as the visible canvas.
- Switching slices must change the visible page cards and details.
- In `交互低保`, each canvas card is already a generated page wireframe result. Page-level actions must continue from that existing page result instead of starting another standalone low-fidelity generation flow.
- Page card quick actions are fixed Agent tool entries: `给布局方案`, `补交互细节`, `重生成本页`. They are capability entry points only; the concrete page plan, interaction details, and regeneration result must still come from the model/backend.
- `给布局方案` should generate layout alternatives for the current page wireframe, not unrelated new pages. Each option should retain the current page's functional outline and key controls while varying product tone, visual weight, layout organization, and information priority.
- Layout candidates may be regenerated by `不满意，重生成 3 个` or the legacy-history affordance `按新规则重生成`; both actions should request a fresh `给布局方案` result for the current page context.
- Applying one layout candidate writes only the selected candidate back to the current page artifact and affected downstream nodes. The other candidates remain part of the Agent conversation/history, not the canvas result.
- Historical persisted labels from older runs, such as `补页面目标`, `补跳转`, `补状态`, `生成低保真`, or `转低保真画布`, must be normalized in the UI to the current page-adjustment/apply model. Users should not see old stage-generation concepts while working on an existing page wireframe.
- Version history should expose generation/source transparency when available, such as `model-generated`, `hydrated`, `cached`, `regenerated`, and the trigger that produced the current low-fidelity output.
- Version history for selected layout candidates should identify the selected option and use a layout-specific source such as `agent-layout-option`.

## Must Not Do

- Do not prefer `totalDesignFlow.pages` over `stageCanvases['interaction-lofi'].nodes` when artifacts exist.
- Do not replace generated wireframes with plain page summaries.
- Do not treat generated gray low-fidelity images as a replacement for the required ASCII/text layout diagram in the page interaction document.
- Do not import Draw.io main-flow or state-machine files as ordinary `interaction-lofi` page nodes; they are global file artifacts and may only be referenced from the page interaction document or Agent artifact cards.
- Do not replace the fullscreen wireframe with page-level interaction instructions; expose interaction instructions as a secondary view.
- Do not render `低保原型` from frontend-hardcoded business screenshots or product examples. It must derive from `pageLayoutArtifact` fields such as `asciiWireframe`, `prototypeSections`, `layoutRegions`, or `regions`.
- Do not let fallback layout heuristics override model-provided `pageLayoutSpec`, `layoutPlan`, or existing `pageLayoutArtifact`.
- Do not let fallback layout candidates detach from the selected page's existing framework. Fallback is allowed only to repair weak candidate quality while preserving the current page outline.
- Do not hide the fullscreen wireframe while keeping only route or state summaries.
- Do not render another separate structure tool instead of the actual Agent/canvas result.
- Do not put model/business conclusions directly into fixed page-card buttons. Buttons launch the Agent workflow; generated content belongs in model/backend artifacts.
- Do not show `生成低保真` as a page-level next action once `交互低保` already has page wireframes. At that point the user is adjusting current page results, not entering the low-fidelity stage again.

## Regression Checks

- Canvas cards render `.canvas-node-wireframe-preview`.
- Fullscreen renders `.canvas-page-layout-artifact`.
- Fullscreen exposes a page-level low-fidelity prototype tab derived from `pageLayoutArtifact`.
- Current-stage nodes are passed into `WorkflowCanvasPage`.
- Fallback nodes are used only when the active slice has no matching stage nodes.
- Interaction page cards keep the three fixed tool entries and do not regress to generic `补页面目标` / `补跳转` / `补状态` actions.
- Old run quick replies and page actions are normalized so existing historical runs immediately follow the current interaction model without requiring regeneration.
- `给布局方案` returns three comparable options for the current page, supports old-rule regeneration, and applying an option refreshes the canvas with selected-option loading and history metadata.
