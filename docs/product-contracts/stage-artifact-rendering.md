# Stage Artifact Rendering Contract

## Goal

The workflow canvas uses a stage artifact rendering model. Backend/model artifacts are the source of truth. Frontend renders those artifacts in the workflow canvas without inventing business page content or replacing the current implementation with a separate diagram tool.

## Current Implementation Stack

- Frontend: Vue 3 + Vite.
- Workflow canvas: DOM cards, CSS layout/scale, and SVG edges inside `WorkflowCanvasPage.vue`.
- Stage data: `totalDesignFlow.stageCanvases`.
- Interaction lofi wireframes: backend/model `pageLayoutArtifact` rendered as ASCII in `<pre>`.
- UI visual stage: `visualPreview`, `visualBrief`, `generationActions`, `targetGenerator`, and `artifactStatus` on `ui-visual` stage nodes.
- Image generation target: `gpt-image-2` when configured through the image provider.

## Must Preserve

- Stage canvas nodes are the canonical source for generated stage artifacts. Props/root canvas nodes are display fallbacks only when the active stage has no stage canvas nodes.
- `requirement-dissection` stage nodes should come from `requirementDissectionArtifact.productAnalysisPipeline.tabs`, not from frontend-authored business cards.
- Requirement-dissection tab fullscreen details should render `detailBlocks` generically and resolve `sourceRef` from canonical artifact fields such as `pageCoverageMatrix`, `decisionPointMatrix`, `exceptionRecoveryMatrix`, `dataFlowGraph`, `stateMachineMap`, `featureJumpGraph`, and `dataSharingMechanism`.
- The Advanced UX `整体交互链路` tab should render top navigation, page hierarchy, user journey, data flow, state machine, and cross-function relations as distinct detail blocks without duplicating the underlying fields.
- Top navigation and page hierarchy detail blocks should render backend-owned page bindings such as `navigationItems.targetPageId`, `activeState`, `visibilityRule`, `pageId`, `level`, and `pageType` when available.
- `analysisGuidance` is backend/model guidance only. It must not be rendered in the requirement-dissection canvas detail, Agent drawer, or any second report grid.
- `interaction-lofi` nodes must keep `node.pageLayoutArtifact` so card preview, fullscreen detail, and Agent context read the same backend/model-owned structure.
- `pageLayoutArtifact.asciiWireframe` is generated by `page-layout-artifact-renderer` and `diagram-generator`, then rendered by the frontend with `<pre>`.
- Model-provided `pageLayoutSpec`, `layoutPlan`, `structuredLayoutPlan`, or `page_layout_spec` is the preferred page layout source. Fallback layout heuristics only keep legacy or sparse runs renderable.
- UI visual nodes are generation targets derived from the page flow. They show preview/status/actions first; actual images are generated later through `generationActions` and patched back onto the same stage node.
- UI visual generation should preserve target aspect ratio metadata such as `aspectRatio`, `targetAspectRatio`, or `screenRatio` through request payload, generated `artifact`, `visualPreview`, canvas cards, fullscreen detail, and Agent image result cards.
- When UI visual generation has no explicit ratio, app/mobile/mini-program surfaces should default to `375x812` (`375:812`), and web/desktop/admin surfaces should default to `1920x1080` (`1920:1080`). The target size is generation metadata, not a frontend-authored business artifact.
- Project-scoped UI visual generation may include project knowledge/prototype screenshots as `projectVisualContext` for the image model. This context is allowed only when the active run's `demandScope === 'project'`; non-project runs must not inherit current URL/project knowledge.
- Generated image display should preserve the original image ratio. Once an image loads, canvas cards, fullscreen detail, and Agent image result cards should prefer the image's natural pixel ratio over target ratio metadata such as `375:812` or `1920:1080`; target metadata is generation intent, not a stretching rule. Agent image result cards use bounded width/height and proportional `contain` behavior instead of fixed-height cropping.
- While UI visual generation is running from Agent/canvas actions, the Agent conversation should show an image-shaped generating placeholder for the target node. When generation succeeds, the result card must use the backend-returned generated node before falling back to local canvas state.
- If the backend returns a generated node before persisted image URLs are fully reflected in that node, the frontend must build the Agent result card from the current hydrated canvas node in `totalDesignFlow.stageCanvases` so the Agent image card and canvas image never diverge.
- Reopening a generated UI visual canvas node should reconcile the Agent session from that node's `visualPreview` or `artifact` image URL when a historical Agent result message is missing the URL.
- HTML/Vue generated artifacts should stay as backend/model-owned code artifacts on the node (`codePreview` or `artifact`) and render in Agent through the shared markdown/code-card path, not a separate artifact renderer.
- Generation results must update the same node across root canvas and `totalDesignFlow.stageCanvases` so canvas cards, fullscreen detail, version history, and Agent context stay aligned.
- Agent page actions such as `给布局方案`, `补交互细节`, and `重生成本页` are capability entry points. They send the current stage, page node, slice, and artifact context to Agent/model; concrete page changes still come from backend/model output and are applied back to the current artifact.
- Layout type terms such as waterfall, card flow, list flow, information flow, left-image-right-text, image-top-text-bottom, form flow, checkout flow, two-column menu, and grid entry are model output candidates, not frontend-hardcoded page content.

## Must Not Do

- Do not replace the workflow canvas with Mermaid, ReactFlow, AntV X6, Excalidraw, HTML Canvas, or a separate diagram workbench unless the product contract is intentionally redesigned.
- Do not reconstruct business page layout in the frontend when `pageLayoutArtifact` exists.
- Do not use `totalDesignFlow.pages` or generic page summaries to overwrite existing `stageCanvases[stageId].nodes` artifacts.
- Do not render both the requirement pipeline detail and the legacy all-in-one requirement report for the same run.
- Do not copy root requirement matrices into frontend-only display fields. Use `detailBlocks[].sourceRef` and canonical backend artifact fields.
- Do not hardcode tea-drink, podcast, or other business pages in frontend UI to pretend a model generated them.
- Do not let fallback presets or heuristics override model-provided stage canvas artifacts.
- Do not generate fake UI visual images when no image provider is configured; keep the node in `待配置` or `待生成` state.
- Do not show image prompts, screen contracts, or project visual context as default Agent chat prose after high-fidelity image generation; keep them as generation basis metadata or collapsed evidence.
- Do not send project knowledge/prototype screenshots into image generation for non-project workflow runs.
- Do not stretch, crop, or force generated app/web screenshots into a fixed-height Agent image box.
- Do not leave the Agent image result area blank during visual artifact generation.
- Do not persist an Agent "generated image" success message with an empty image URL when the canvas node already has a generated image URL.

## Ownership

- Backend owns `totalDesignFlow`, `stageCanvases`, `pageLayoutArtifact`, visual artifacts, generation actions, and artifact persistence.
- Frontend owns rendering, stage/slice selection, card layout, fullscreen detail, Agent entry points, visual pending/generated states, modal layering, and fallback display.
- Agent owns continuation and refinement of the selected node context; it is not a second standalone page generator.

## Regression Checks

- `interaction-lofi` stage cards render `pageLayoutArtifact.asciiWireframe`.
- Fullscreen detail renders the same `pageLayoutArtifact`.
- Slice filtering keeps backend artifact nodes and falls back to page-derived nodes only when the active slice has no artifact nodes.
- Requirement-dissection tests cover the ordinary nine-chapter pipeline, the advanced UX ten-node variant, sourceRef resolution, page frame contracts, and guidance single-display behavior.
- `ui-visual` stage nodes render pending/configuration/generated visual states from `visualPreview` and `artifact`.
- `generationActions` route artifact generation and patch the result back into `totalDesignFlow.stageCanvases`.
- Project UI visual generation carries target ratio and project visual reference context into the image-provider prompt/context, while generated image cards show compact ratio metadata.
- App/web UI visual generation carries inferred target image size when explicit ratio metadata is absent, and Agent/fullscreen image rendering preserves image proportions.
- Agent visual generation pending and success states are both rendered as visual artifact cards: pending uses a generating placeholder, success uses the generated image URL/data URL.
- Opening a generated UI visual node repairs empty historical Agent image messages from the canvas node and persists the repaired session.
- HTML/Vue node generation routes through `generationActions` and appends fenced code to the same Agent conversation.
- Agent actions remain capability entries and do not contain fixed business output.
- Ordinary text remains prose; only explicit artifacts are rendered as artifact cards.
