# 流程通 Phase 3：Skill 工作台信息架构与交互方案生成设计

## 背景

当前版本已经具备 Skill 草稿、默认通用 Skill、完整流程运行器、运行记录和输出资产的基础能力，但这些能力都堆在一个 `Skill 工作台` 页面里。用户看到的是多个模块同时出现：完整 Skill 流程、项目诊断台、执行结果、运行记录、输出资产、Skill 中心、Skill 编辑器。结果是系统看起来“有功能”，但主流程不清楚，用户不知道下一步该点哪里。

Phase 3 的目标不是继续增加更多 Skill，而是把已有能力拆成可使用的信息架构，并新增一个成熟的 `交互方案生成 Skill`，让用户可以从一句模糊需求进入一个完整、可点击、可保存的产品/交互设计流程。

## 产品目标

- 把“输入需求 → 诊断 → 推荐 Skill → 分步执行 → 保存资产 → 查看版本”的主流程跑通。
- 把运行记录从主页面移走，只作为资产版本、最近活动或详情抽屉出现。
- 把 Skill 中心从执行页面拆出去，避免用户在工作流中被编辑器打断。
- 新增 `交互方案生成 Skill`，覆盖信息架构、任务流、页面结构、组件状态和交互评审。
- 将 `zephyrwang6/pm-skills` 这类外部资料定位为“方法库来源”，而不是直接当成线上可执行代码。

## 范围

### 本阶段包含

- 新的信息架构与导航。
- 独立的项目诊断页。
- 独立的流程运行页。
- 独立的资产库页。
- 独立的 Skill 中心页。
- 新增内置 `交互方案生成 Skill`。
- 将运行记录隐藏到资产详情或最近活动入口。
- 保留用户 Skill 的表单/Markdown 双模式编辑。

### 本阶段不包含

- 真正接入远程大模型服务。
- 真正联网安装 GitHub Skill。
- 多用户权限、团队协作、云端同步。
- Figma 自动生成高保真稿。
- 完整 RAG 向量检索。

## 外部 Skill 策略

`zephyrwang6/pm-skills` 可以作为产品经理方法库参考，但不能直接决定系统形态。

处理策略：

- 短期：把它当成“外部 Skill 草稿来源”。用户可以粘贴 Markdown 或链接，系统导入为待审核草稿。
- 中期：提供“方法库导入器”，把外部 Skill 转换为系统 Skill 结构：适用场景、输入要求、知识范围、执行步骤、输出格式、质量检查。
- 长期：支持安全的 Skill Marketplace，但只能导入声明式 Skill，不执行外部脚本。

判断原则：

- 有明确输入、步骤、输出和质量标准的内容，可以转成系统 Skill。
- 只有角色设定或长 Prompt 的内容，只能作为草稿参考。
- 包含脚本或自动化代码的仓库，只允许只读解析，不能直接执行。

## 信息架构

主导航改为：

1. `项目诊断`
2. `流程运行`
3. `资产库`
4. `Skill 中心`
5. `网页工厂`
6. `资料库`
7. `设置`

旧的 `PM Skill` 和 `UX Skill` 页面在 Phase 3 中不作为主入口突出展示。它们的价值会被并入 `项目诊断`、`流程运行` 和 `Skill 中心`。

### 项目诊断

用户首次进入 Skill 相关能力时默认看到这里。

页面职责：

- 输入原始需求。
- 选择当前项目或知识库范围。
- 使用默认通用 Skill 进行需求诊断。
- 判断需求清晰度。
- 推荐 1-3 个可执行 Skill。
- 展示最近活动的简短入口。

页面不展示：

- 完整运行记录列表。
- 输出资产全文。
- Skill 编辑器。

推荐 Skill 卡片应该提供明确动作：

- `开始流程`
- `查看为什么推荐`
- `切换 Skill`

### 流程运行

流程运行页负责让用户完成一个 Skill 的完整步骤。

页面结构：

- 左侧：步骤导航。
- 中间：当前步骤输入表单。
- 右侧：当前步骤输出、质量检查、确认按钮。
- 顶部：当前流程名称、状态、保存资产、导出 Markdown。

每个步骤必须具备：

- 步骤目标。
- 必填字段。
- 可选参考信息。
- 生成结果按钮。
- 可编辑输出。
- 质量检查。
- 确认进入下一步。

用户不能在缺少关键输入时直接进入下一步。系统可以允许用户“以假设继续”，但必须把假设写入输出资产。

### 资产库

资产库负责承载最终产物，而不是让执行页面无限增长。

资产类型：

- 需求诊断卡
- 需求拆解文档
- 用户旅程地图
- 交互方案
- PRD
- 竞品分析
- 评审清单

资产详情包含：

- 标题
- 类型
- 来源 Skill
- 创建时间
- 原始输入
- 最终输出
- 使用的步骤
- 运行记录入口
- 版本历史

运行记录只在这里通过“查看运行链路”出现，不在主工作台铺开。

### Skill 中心

Skill 中心负责管理 Skill，不参与当前流程执行。

功能：

- 查看系统 Skill。
- 复制系统 Skill 为我的 Skill。
- 新建我的 Skill。
- 表单编辑。
- Markdown 编辑。
- 外部链接或 Markdown 导入为待审核草稿。
- 结构校验。
- 启用/停用用户 Skill。

Skill 中心的默认列表应按照类型分组：

- 默认通用
- 需求理解
- 用户研究
- 产品方案
- 交互设计
- 交付验证
- 我的 Skill

## 新增内置 Skill：交互方案生成

### 定位

`交互方案生成 Skill` 面向产品交互设计场景，输入可以是模糊需求、已有流程、页面问题、B 端业务规则或改版目标。它不直接输出一个泛泛的设计建议，而是把需求转成可执行的页面结构、流程和交互说明。

### 适用场景

- 需要从需求描述生成交互方案。
- 当前页面或流程很乱，需要改版。
- B 端后台、审批、配置、表单、数据看板等复杂系统。
- 需要给开发或设计团队一份可评审的交互说明。

### 不适用场景

- 只需要视觉风格稿。
- 只需要一段营销文案。
- 没有任何业务目标或用户目标，且用户拒绝补充假设。

### 步骤

1. `需求诊断`
   - 明确用户、目标、业务背景、成功标准、约束和缺失信息。
   - 输出：需求诊断卡。

2. `信息架构`
   - 拆分对象、导航、页面层级、内容分组和权限入口。
   - 输出：信息架构建议。

3. `核心任务流`
   - 梳理主路径、分支路径、回退路径和关键决策点。
   - 输出：任务流说明。

4. `页面结构`
   - 定义每个关键页面的模块、主次动作、输入输出和跳转关系。
   - 输出：页面结构表。

5. `组件状态与异常`
   - 补充表单、列表、详情、弹窗、空状态、错误状态、加载状态、权限状态。
   - 输出：状态与异常清单。

6. `交互评审`
   - 用一致性、效率、可理解性、可恢复性、可测试性检查方案。
   - 输出：评审问题与修改建议。

7. `交付拆解`
   - 转成设计任务、开发任务、验收标准和后续风险。
   - 输出：交付清单。

### 输出资产

最终资产类型为 `交互方案`，包含：

- 背景与目标
- 用户与场景
- 信息架构
- 核心任务流
- 页面结构
- 组件状态
- 异常流程
- 交互评审
- 开发任务拆解
- 验收标准
- 假设与待确认问题

## 数据模型调整

现有 `state.skillRuns` 和 `state.assets` 继续保留，但展示位置改变。

新增或补强字段：

- `activeWorkflowRun.lastSavedAssetId`
- `asset.workflowId`
- `asset.skillId`
- `asset.runId`
- `asset.versions`
- `asset.hiddenRunRecords`
- `workflow.recommendedFor`

不需要在本阶段引入后端数据库。继续使用当前前端状态和本地持久化方式即可。

## 页面状态

### 项目诊断状态

- 空状态：提示用户输入需求。
- 诊断中：显示当前正在读取的知识范围。
- 信息不足：显示最多 3 个澄清问题。
- 可执行：显示推荐 Skill。
- 已开始：跳转到流程运行页。

### 流程运行状态

- 未开始：等待用户选择流程。
- 运行中：当前步骤可输入、生成和确认。
- 待补充：必填字段缺失。
- 待确认：已生成输出，等待用户确认。
- 已完成：可以保存资产、导出 Markdown、进入资产库。

### 资产状态

- 草稿
- 已保存
- 已更新
- 已导出

## 错误处理

- GitHub Skill 安装失败：显示为“无法联网安装，可粘贴 Markdown 导入草稿”。
- Skill 结构不完整：不能启用，只能保存为草稿。
- 流程步骤缺少必填项：阻止确认进入下一步。
- 输出为空：不允许保存资产。
- 重复保存：生成新版本，不覆盖旧资产。

## 测试策略

### 单元测试

- 新增 `交互方案生成 Skill` 的 workflow 定义存在。
- 每一步都有 `id`、`title`、`goal`、`requiredFields` 和 `outputTitle`。
- `createWorkflowRun` 能创建完整步骤状态。
- `produceStepOutput` 能为交互方案流程生成对应输出。
- `completeCurrentStep` 能逐步推进到完成。
- `exportWorkflowRunMarkdown` 包含全部步骤输出。

### 浏览器验证

- 项目诊断页默认可见。
- 推荐 Skill 后可以进入流程运行页。
- `交互方案生成 Skill` 可以完整点击至少前两步。
- 主页面不再直接展示完整运行记录列表。
- 资产库能看到保存后的资产。
- Skill 中心仍能表单/Markdown 编辑 Skill。

## 实施顺序

1. 调整导航和视图拆分。
2. 把旧 `Skill 工作台` 拆到 `项目诊断`、`流程运行`、`资产库`、`Skill 中心`。
3. 新增 `交互方案生成 Skill` workflow。
4. 隐藏主页面运行记录，只保留最近活动入口。
5. 补充资产详情与运行链路入口。
6. 增加 workflow 测试。
7. 运行 `npm test` 和 `npm run build`。
8. 用浏览器验证完整点击流程。

## 已确认边界

- Phase 3 先不做真实 GitHub 安装；若后续网络恢复，再接入外部 Skill 导入器。
- Phase 3 先不做真正 RAG；知识库读取先使用已有前端资料对象。
- Phase 3 不处理多项目复杂权限，继续以单用户本地原型为主。
