# 总流程设计方案工作台

## 背景

当前设计方案入口以多个 Skill 为主：UX 设计确认、对话 Skill、智能推荐、交互方案生成、竞品监控等。这个结构的问题是用户要先理解工具，再完成方案；而用户真正想做的是从需求或文档出发，一路得到需求确认、页面级交互低保、UI 视觉、HTML、Vue 和知识沉淀。

因此主产品心智改为一个独立的“总流程”工作台。小 Skill 不再作为主入口，而是退到后台成为阶段能力、局部按钮或高级工具。

## 目标

- 用户只需要输入需求、上传文档或选择项目知识，就能进入统一设计方案流程。
- 大文档先被拆成小需求切片，不再被局部关键词误判成单一主意图。
- 每个小需求可以生成自己的流程画布、页面节点、交互说明、UI 方向和代码交付。
- 每个阶段的详情模板固定，内容和画布结构由 AI 生成，系统负责校验和补缺提示。
- 小 Skill 不再打断主流程，只作为后台能力帮助某个阶段完成专项分析。

## 非目标

- 不在主入口继续强化 Skill 下拉选择。
- 不要求第一版就完成高保真设计工具或完整 Vue 工程生成。
- 不把竞品监控做成总流程的强制步骤，它是证据能力，不是主路径。
- 不让前端拼接关键业务结论，核心结构应由后端 AI/分析器返回。

## 产品原则

1. 用户在走设计方案，不是在选择工具。
2. 先解剖文档，再判断意图。
3. 一个文档可以拆成多个小需求，一个小需求可以有多条流程。
4. 一个页面可以被多条流程引用，但页面详情只维护一份。
5. 阶段固定，阶段详情 Schema 固定，节点数量和连线由 AI 生成。
6. 每次转阶段都保留版本，用户可以回看、对比和继续提问。

## 总流程阶段

```text
输入需求 / 上传文档 / 选择项目知识
-> 需求分析
-> 小需求切片
-> 缺口确认
-> 交互低保
-> UI视觉
-> HTML
-> Vue
-> 验收沉淀
```

### 需求分析

目标：读懂输入材料，不急着生成页面。

AI 输出：

- 需求原文
- 文档摘要
- 产品定位
- 目标用户
- 核心业务对象
- 业务模块
- 页面/流程/接口/状态/权限/商业化内容
- 项目知识库引用
- AI 推断
- 待确认风险
- 来源段落

验收：

- 完整产品文档不能被“登录注册”等局部词抢走主意图。
- 必须区分事实、推断和待确认。
- 必须保留来源引用，方便用户知道 AI 为什么这样判断。

### 小需求切片

目标：把复杂文档拆成左侧可管理的小需求。

小需求字段：

- id
- 名称
- 一句话目标
- 来源段落
- 涉及用户
- 涉及业务对象
- 涉及页面数量
- 优先级 P0/P1/P2
- 当前阶段状态
- 待确认问题数量
- 是否阻塞后续阶段

示例：Podcastor 文档应拆成：

- 首页与创作入口
- 脚本生成流程
- 音频播客生成流程
- 视频播客制作流程
- 主持人/音色/背景资产库
- 作品生成与存储
- 短视频切片与分发
- 登录注册与商业化
- 管理端与平台分发

### 缺口确认

目标：让用户知道哪些信息会影响方案继续生成。

每个小需求展示：

- 已确认事实
- AI 推断
- 待确认问题
- 不确认会影响什么
- 建议问谁
- 是否阻塞下一阶段

规则：

- 只问会改变方向、范围、流程、接口或验收的问题。
- 不要把所有小问题都丢给用户。
- 用户回答后必须能刷新当前小需求，而不是重开一份无关分析。

### 交互低保

目标：回答页面怎么走、流程怎么串、异常怎么回。

中间画布展示：

- 流程泳道
- 页面节点
- 页面之间的连线
- 异常分支
- 返回路径
- 并行流程

页面节点详情固定为：

- 页面目标
- 核心模块
- 用户点击动作
- 跳转到哪个页面
- 加载 / 空 / 失败 / 权限状态
- 接口或数据依赖
- 验收点

节点展开规则：

```text
脚本生成页
├─ 输入 Prompt
├─ 粘贴 URL
├─ 上传 PDF/PPT
├─ 生成中
├─ 生成成功
├─ 生成失败
└─ 内容违规
```

连线类型：

- 主路径：实线
- 分支路径：虚线
- 异常路径：警示线
- 返回路径：回折线
- 并行路径：分叉线

### UI视觉

目标：把交互低保升级成页面级视觉方向和组件设计说明。

每个页面输出：

- 视觉风格
- 信息层级
- 组件清单
- 页面状态样式
- 设计变量
- 页面级 UI 方向

设计变量包括：

- 主色
- 辅助色
- 文字层级
- 圆角
- 间距
- 阴影
- 按钮高度
- 卡片密度

视图：

- 页面视觉总览
- 单页面视觉详情
- 组件复用关系
- 状态样式总览

### HTML

目标：把已确认页面方案变成可运行 HTML Demo。

输出：

- 页面结构
- 样式文件
- 交互脚本
- Mock 数据
- 可运行预览
- 代码说明

### Vue

目标：把 HTML/视觉方案拆成可维护 Vue 工程。

输出：

- 页面路由
- 组件拆分
- 状态管理
- Mock 数据
- 接口契约
- 文件结构
- 运行方式
- 预览入口

Vue 示例：

```text
src/pages/HomeEntry.vue
src/pages/ScriptGenerator.vue
src/pages/AudioPodcast.vue
src/pages/VideoPodcast.vue
src/components/UploadPanel.vue
src/components/GenerationProgress.vue
src/components/AssetPicker.vue
src/services/api.ts
src/mocks/podcastor.ts
```

### 验收沉淀

目标：把本次方案沉淀为下次可复用的项目知识。

沉淀内容：

- 最终需求切片
- 页面流程图
- 页面级交互低保
- UI 规则
- 接口契约
- 验收清单
- 设计决策
- 项目知识库条目
- 版本记录
- 下次复用建议

下次同项目需求进入时，系统应能判断：

- 这是新增入口
- 这是修改旧流程
- 这是影响已有页面
- 这是复用已有接口
- 这是新增知识库规则

## 页面布局

```text
顶部：总流程阶段导航
左侧：小需求切片列表
中间：当前小需求的 Agent 工作台 / 画布 / UI / 代码预览
底部：当前画布的页面节点导航
右侧：详情面板 + Agent
```

### 左侧：小需求切片

左侧不放页面节点，而放需求切片：

```text
全部需求切片
- 首页与创作入口
- 脚本生成流程
- 音频播客生成流程
- 视频播客制作流程
- 资产库复用
```

点击小需求后，中间区域切换到该小需求的当前阶段内容。

### 中间：阶段主视图

不同阶段中间区域不同：

- 需求分析：Agent 工作台，确认输入、文档、项目背景、事实/推断/风险。
- 小需求切片：Agent 工作台，确认切片边界、合并拆分和优先级。
- 缺口确认：Agent 工作台，确认会影响范围、流程、接口或验收的问题。
- 交互低保：页面流程画布。
- UI视觉：页面视觉总览。
- HTML：HTML 运行预览和源码。
- Vue：Vue 运行预览和工程代码。
- 验收沉淀：验收和知识沉淀总览。

### 底部：页面节点导航

底部只显示当前小需求下的页面节点：

```text
首页入口 -> 输入方式 -> 脚本生成 -> 脚本预览 -> 编辑确认 -> 生成音频
```

作用：

- 快速定位当前画布页面。
- 节点多时横向滚动。
- 点击后中间画布聚焦对应页面。
- 显示页面状态数量和是否有待确认项。

### 右侧：详情 + Agent

右侧详情根据阶段切换模板。Agent 永远围绕当前小需求、当前阶段、当前节点回答。

Agent 输入上下文包括：

- 原始输入
- 当前小需求
- 当前阶段
- 当前节点
- 已确认问题
- 关联项目知识
- 关联竞品证据
- 版本历史

## 小 Skill 的新定位

小 Skill 不作为主流程入口。

| 旧 Skill | 新定位 |
| --- | --- |
| 对话 Skill | 右侧 Agent 的自由讨论能力 |
| 智能推荐 Skill | 需求分析阶段的内部路由能力 |
| UX 设计确认 Skill | 缺口确认阶段的内部确认能力 |
| 交互方案生成 | 交互低保阶段的内部生成能力 |
| 竞品监控 Skill | 给小需求或页面节点补证据 |
| PRD/交互文档 | 验收沉淀里的导出能力 |
| HTML | 交付阶段能力 |
| Vue | 工程化交付阶段能力 |

用户看到的是动作，不是 Skill 名称：

- 补竞品证据
- 补异常状态
- 生成 UI 方向
- 转 HTML
- 转 Vue
- 沉淀知识库

## 统一数据结构

```json
{
  "runId": "",
  "currentStage": "interaction-lofi",
  "requirementSlices": [],
  "activeSliceId": "",
  "flows": [],
  "pages": [],
  "pageStates": [],
  "edges": [],
  "visualSpecs": [],
  "codeArtifacts": [],
  "details": {},
  "knowledgeRefs": [],
  "knowledgeDeposits": [],
  "versions": []
}
```

字段说明：

- requirementSlices：左侧小需求。
- flows：每个小需求下的流程。
- pages：页面节点，页面可被多条流程引用。
- pageStates：页面内状态、异常、权限、加载、空态。
- edges：页面或状态之间的连线。
- visualSpecs：UI 视觉规范。
- codeArtifacts：HTML/Vue/Mock/API 文件。
- details：右侧详情内容。
- knowledgeRefs：读取到的项目知识引用。
- knowledgeDeposits：本次沉淀的知识条目。
- versions：每次生成或修改的版本快照。

## AI 输出协议

AI 可以决定：

- 小需求数量
- 小需求名称
- 每个小需求的流程数量
- 页面节点数量
- 页面节点名称
- 分组和连线
- 页面状态
- 详情内容

系统固定校验：

- 是否有小需求
- 是否有来源引用
- 每个页面是否有目标、模块、动作、跳转、状态、接口、验收点
- 是否区分事实、推断、待确认
- 是否有版本快照
- 是否有下一步动作

如果缺失，系统不伪造结论，而是提示需要补齐或重新生成。

## 两个样例

### 茶饮小程序

输入：

```text
做一个茶饮小程序，需要首页选店、取餐方式、菜单商品、商品定制、购物车结算、支付下单、取餐通知、会员优惠。
```

预期：

- 小需求数量：1 个。
- 小需求名称：茶饮点单主流程。
- 页面节点：
  - 首页与选店
  - 取餐方式
  - 菜单与商品
  - 商品定制
  - 购物车与结算
  - 支付与订单生成
  - 制作与取餐通知
  - 会员与优惠
  - 前后端与验收

### Podcastor 文档

输入：

```text
Podcastor.ai v1.0 产品文档
```

预期：

- 产品总意图：零门槛 AI 播客工作站。
- 不允许被“注册/登录”局部词识别为认证弹窗。
- 小需求切片：
  - 首页与创作入口
  - 脚本生成流程
  - 音频播客生成流程
  - 视频播客制作流程
  - 主持人/音色/背景资产库
  - 作品生成与存储
  - 短视频切片与分发
  - 登录注册与商业化
  - 管理端与平台分发

## 第一版实施范围

第一版先做闭环骨架，不做所有高级能力：

1. 新增总流程主入口，默认替代 Skill 选择入口。
2. 后端新增总流程分析结果结构。
3. 文档分析先返回需求分析和小需求切片。
4. 左侧改为小需求列表。
5. 中间保留现有画布能力，但按 activeSliceId 过滤显示。
6. 底部新增当前小需求的页面节点导航。
7. 右侧详情按阶段切换模板。
8. 茶饮和 Podcastor 两个样例通过回归测试。

后续增强：

- 更真实的可点击原型预览能力。
- Figma 页面生成与回写。
- 更完整的 HTML/Vue 工程文件生成。
- 知识库沉淀自动分类与复用。

## 验收清单

- [x] 设计方案入口默认进入总流程，而不是要求用户选择小 Skill。
- [x] 上传完整产品文档后，系统先拆小需求，不直接输出单一页面方案。
- [x] Podcastor 文档不会被识别成认证弹窗。
- [x] 茶饮小程序能生成单一小需求和 9 个页面节点。
- [x] 左侧展示小需求切片，不展示页面节点。
- [x] 底部展示当前小需求下的页面节点导航。
- [x] 交互低保页面节点包含页面目标、核心模块、动作、跳转、状态、接口、验收点。
- [x] 右侧 Agent 能拿到当前阶段、当前小需求和当前节点上下文。
- [x] 转阶段或重新生成会保存版本。
- [x] 小 Skill 不再作为主流程心智出现，只作为后台能力或局部动作。
