# Admin Resource Ownership Design

## 背景

侧边栏当前有项目空间、设计方案、工程开发、知识库、交付资产、技能中心、系统设置。此前后台管理台已经覆盖项目、资料、资产、工程页面和工作流运行，但技能中心和系统设置还主要停留在前端状态或本地配置里。

## 结论

推荐采用“后端作为业务资源唯一真源，前端只保存临时 UI 状态”的方案。

后端接管可沉淀、可审计、可共享的数据：项目、知识资料、交付资产、工程页面、工作流运行、Skill 定义、系统配置。前端保留当前选中项、弹窗开关、临时草稿、预览缩放、上传过程这类交互状态。后台管理台只做内部治理、排查、审计和安全操作，不替代前台产品工作流。

## 方案对比

1. 前端继续本地存储：开发最快，但数据分散，后台看不到技能和配置，后续多人协作、审计、迁移都会困难。
2. 全量立刻迁移到后端：架构最干净，但会一次性改动 Skill 编辑器、运行器、设置页和兼容逻辑，风险较高。
3. 渐进式后端接管：先在后端建立 `skills`、`settings` 资源和后台 CRUD，再逐步让前台读写这些接口。推荐此方案，风险低、可验证、符合大厂常见的资源治理节奏。

## 前后端分工

前端负责：页面布局、交互状态、表单草稿、上传入口、调用 API、展示后端结果。

后端负责：资源模型、持久化、业务规则、文件解析、网页采集、AI/Agent 调用、错误格式和恢复建议。

后台负责：资源列表、搜索分页、新增编辑删除、批量删除、导出、系统诊断、API 调试、配置和 Skill 治理。

## 数据流

前台创建或修改业务资源时调用 `/api/workspace/*` 或后续专用资源接口；后端写入 workspace store；后台通过 `/api/admin/records/:collection` 管理同一份数据；前台启动时通过 `/api/workspace` 合并后端快照。

## 本轮落地

后端 workspace store 新增 `skills` 和 `settings` 集合，后台管理台新增技能管理和系统配置模块，支持搜索、分页、详情、新增、编辑、单条删除和批量删除。
