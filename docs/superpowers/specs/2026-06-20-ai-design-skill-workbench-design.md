# AI 产品/交互设计 Skill 编排工作台设计

## 产品定位

流程通是一个面向产品经理、交互设计师和体验设计师的 AI Skill 编排工作台。它不把 AI 定位成一个只会聊天的通用助手，而是让 AI 读取项目知识库、项目流程、历史需求、竞品资料和设计规范，再通过默认通用 Skill 与用户自定义 Skill 组合，产出可追溯、可复用、可沉淀的产品与交互设计资产。

核心目标是解决三类常见工作：

- 新项目从 0 到 1 梳理：从模糊想法、业务目标和零散资料中形成产品定义、用户路径、PRD 和初步交互方案。
- 已有项目改版优化：读取现有页面、流程、竞品、用户反馈和历史决策，识别体验问题并生成优化方案。
- B 端复杂系统设计：梳理角色、权限、状态、审批、表单、数据看板和异常流程，降低复杂业务转化为界面方案的难度。

模糊需求是主流程的一部分，而不是异常情况。系统默认先做需求诊断和上下文补齐，再决定是否生成方案。

## 设计原则

- 先理解，再生成：用户输入不清晰时，系统先澄清问题，不急于产出完整方案。
- 先读知识库，再推断：输出必须优先引用项目已有资料，AI 推断需要显式标注。
- Skill 是工作协议，不只是提示词：每个 Skill 必须包含适用场景、输入要求、知识库读取规则、追问规则、工作步骤、输出格式和质检标准。
- 默认通用 Skill 负责兜底和路由：用户不选 Skill 也能开始；用户选择自定义 Skill 时，默认 Skill 仍负责前处理。
- 用户 Skill 可扩展但要安全：外部导入只能作为待审核草稿，不能直接执行脚本或获得系统权限。
- 结果要沉淀为项目资产：每次执行需要保存输入、引用资料、使用的 Skill、输出结果、版本和后续动作。

## 目标用户

### 产品经理

需要把业务想法、用户反馈、竞品变化和团队讨论转成 PRD、用户故事、MVP 范围、优先级和验收标准。

### 交互设计师

需要把需求、流程和业务规则转成信息架构、任务流、页面流程、交互说明、异常状态和设计评审材料。

### 体验设计师/UX 研究员

需要整理用户画像、用户旅程、访谈记录、痛点机会、可用性测试计划和体验优化建议。

### 设计负责人/产品负责人

需要沉淀团队方法论，把团队自己的设计流程、评审标准和交付模板做成可复用 Skill。

### Skill

Skill 是一个可执行的产品/交互设计工作协议。

Skill 分为三类：

- 默认通用 Skill：系统内置，始终存在，不能删除。
- 系统 Skill：系统预置的经过审核的方法论 Skill。
- 我的 Skill：用户通过表单、Markdown 或外部导入创建的自定义 Skill。

## 默认通用 Skill

默认通用 Skill 是系统的总调度员。它不替代所有专业 Skill，而是负责理解、诊断、检索、路由和前处理。

### 输入

- 用户原始需求
- 当前项目
- 项目知识库
- 项目流程
- 历史需求
- 竞品资料
- 设计规范
- 用户选择的 Skill，可选

### 工作步骤

1. 判断需求清晰度：模糊想法、初步需求、明确需求、待交付任务。
2. 判断任务场景：0 到 1、新项目定义、已有项目优化、B 端复杂系统、页面/流程生成、文档生成。
3. 从知识库检索相关资料：业务目标、用户角色、流程、页面、历史决策、竞品、设计规范。
4. 识别缺失信息：用户、目标、流程、约束、成功标准、边界和风险。
5. 如果关键信息不足，提出最多 3 个高价值澄清问题。
6. 如果信息足够，推荐系统 Skill 或用户自定义 Skill。
7. 如果用户已选择 Skill，检查该 Skill 是否适用，并把上下文整理成目标 Skill 需要的输入。
8. 执行目标 Skill，或在没有目标 Skill 时使用默认流程输出初步结果。
9. 标注输出中的信息来源：知识库引用、用户输入、AI 推断。
10. 将结果保存为项目资产，并记录本次执行链路。

### 输出类型

- 需求诊断卡
- 澄清问题
- 推荐 Skill 链
- 初步方案
- 风险与假设
- 下一步动作

### 质量要求

- 不得跳过知识库读取直接生成完整结论。
- 关键信息缺失时必须追问或明确标注假设。
- 不得把 AI 推断伪装成项目事实。
- 不得生成与用户选择 Skill 明显不匹配的输出。

## 系统 Skill 库

系统 Skill 库是内置方法论集合，来源可以参考公开资料、团队经验和用户常用流程，但必须经过产品侧审核后再内置。

### 外部资料使用判断

外部资料不能直接等同于可执行 Skill，需要先判断它是“Skill 结构参考”“方法论模板”还是“Prompt 灵感”。

- `zephyrwang6/pm-skills`：可作为 Skill 结构参考。它的价值在于把 PRD、评审、优先级、路线图、数据分析、实验、埋点、问卷、竞品等 PM 工作拆成独立 Skill。适合借鉴“一个重复工作对应一个可执行 Skill”的产品架构，但不能直接覆盖交互设计场景，需要补充用户旅程、信息架构、任务流、服务蓝图、B 端流程和交互评审。
- `pmframe.works/framework-user-journey-map`：可作为方法论模板。它适合转成系统内置的“用户旅程地图”Skill，重点吸收阶段、触点、行为、情绪、痛点、机会点和设计行动这些结构。
- GitHub 上的 PM/UX prompt 和模板库：多数只能作为灵感或输出格式参考。只有具备适用场景、输入要求、追问规则、知识库读取规则、工作步骤、输出格式和验收标准的内容，才可以进入系统 Skill 草稿。

判断标准：

- 如果资料只有一句角色设定或通用 prompt，不能作为系统 Skill。
- 如果资料有固定方法、输入和输出，可以转成 Skill 草稿。
- 如果资料包含脚本、插件或自动化代码，只能只读分析，不能直接导入执行。
- 如果资料包含成熟模板，但缺少执行步骤，可以作为输出格式或表单字段参考。

## MVP 内置 Skill

MVP 阶段只内置三个高价值 Skill，避免过早扩张。

### 模糊需求澄清

适用场景：

- 用户只有一句粗略想法。
- 需求缺少目标用户、业务目标、成功标准或场景边界。
- 用户不确定该从 PRD、用户旅程、竞品分析还是交互方案开始。

输出：

- 当前理解
- 需求类型
- 已知信息
- 缺失信息
- 关键风险
- 推荐下一步
- 推荐 Skill 链

### 用户旅程地图

适用场景：

- 需要理解用户完成某个目标时的阶段、触点、行为、情绪、痛点和机会点。
- 适合体验优化、新功能规划、服务流程梳理和跨触点问题诊断。

输出格式：

| 阶段 | 用户目标 | 用户行为 | 触点 | 情绪 | 痛点 | 机会点 | 设计建议 |
| --- | --- | --- | --- | --- | --- | --- | --- |

追问规则：

- 如果缺少用户角色或用户目标，先追问。
- 如果知识库已有相关画像或流程，优先引用知识库。

### PRD/交互方案生成

适用场景：

- 已有足够背景，需要形成产品需求、交互方案或开发交接材料。
- 适合从 0 到 1 定义、已有流程优化、B 端复杂系统模块设计。

输出：

- 背景和目标
- 用户和场景
- 问题定义
- 范围和非范围
- 核心流程
- 页面/模块说明
- 状态和异常
- 方案对比
- 验收标准
- 待确认问题

## 用户自定义 Skill

用户自定义 Skill 支持表单模式和 Markdown 高级模式。

### 表单模式

适合初级用户。用户通过结构化字段创建 Skill。

字段包括：

- Skill 名称
- 描述
- 适用场景
- 不适用场景
- 需要输入的信息
- 知识库检索范围
- 工作步骤
- 追问规则
- 输出格式
- 验收标准
- 示例输入
- 示例输出

### Markdown 高级模式

适合高级用户和团队方法论沉淀。Markdown 内容与表单字段互相映射。

示例结构：

```md
# Skill: 用户旅程地图

## 适用场景
用于梳理用户完成某个目标时的阶段、行为、触点、情绪、痛点和机会点。

## 不适用场景
不用于直接生成高保真 UI，不用于替代业务决策。

## 需要输入
- 用户角色
- 用户目标
- 当前流程
- 业务目标
- 已知痛点

## 知识库检索范围
优先读取：
- 项目背景
- 用户画像
- 业务流程
- 历史需求
- 竞品资料

## 工作步骤
1. 判断输入是否足够。
2. 如果缺少用户角色或目标，先追问。
3. 拆解用户旅程阶段。
4. 为每个阶段补充行为、触点、情绪和痛点。
5. 归纳机会点。
6. 输出设计建议和下一步动作。

## 输出格式
| 阶段 | 用户行为 | 触点 | 情绪 | 痛点 | 机会点 | 设计建议 |
| --- | --- | --- | --- | --- | --- | --- |

## 追问规则
如果需求模糊，最多先问 3 个关键问题。
如果知识库已有答案，优先引用知识库，不重复问用户。

## 验收标准
- 输出必须引用项目上下文。
- 不得凭空补充关键业务规则。
- 必须明确哪些信息来自知识库，哪些是 AI 推断。
```

### 表单与 Markdown 同步

- 表单编辑后自动生成 Markdown。
- Markdown 编辑后尝试解析回表单字段。
- 如果 Markdown 中存在无法解析的自定义段落，保留在高级内容区，不丢失。
- 保存前执行结构检查，提示缺失的适用场景、输入要求、输出格式和验收标准。

## 外部 Skill 导入

外部来源包括 GitHub、网页和本地 Markdown 文件。

### 可导入内容

- Markdown Skill 文档
- YAML/JSON Skill 定义
- Prompt 模板
- 方法论模板

### 不可直接执行内容

- Shell 脚本
- Node/Python 脚本
- 浏览器自动化脚本
- 要求读取密钥、系统文件或外发项目资料的指令

### 导入流程

1. 用户输入 GitHub 链接、网页链接或上传文件。
2. 系统抓取只读内容。
3. 系统识别是否包含 Skill 结构。
4. 系统扫描风险指令。
5. 系统转换为待审核 Skill 草稿。
6. 用户确认来源、许可证、更新时间和内容。
7. 用户编辑并保存到“我的 Skill”。

### 安全检查

风险检查项：

- 是否要求忽略系统规则。
- 是否要求读取密钥、token、环境变量或私有文件。
- 是否要求上传、发送或暴露项目资料。
- 是否包含可执行代码。
- 是否缺少适用场景、输入、输出和验收标准。
- 是否鼓励编造事实或绕过用户确认。

外部 Skill 默认只能作为文本规则参与模型上下文，不获得文件系统、网络、浏览器或 API 调用权限。

## 知识库读取机制

### 资料导入

支持用户导入：

- 本地文档
- Markdown
- PRD
- 会议纪要
- 截图
- 网页采集结果
- 竞品链接
- Figma 链接
- 代码仓库摘要

### 知识结构化

导入后系统需要抽取：

- 业务背景
- 用户角色
- 用户目标
- 页面/模块
- 核心流程
- 业务规则
- 痛点
- 约束
- 指标
- 历史决策
- 待确认问题

### 检索策略

默认通用 Skill 或目标 Skill 执行前，系统根据任务类型检索相关知识。

检索优先级：

1. 当前项目强相关资料。
2. 用户明确上传或选中的资料。
3. 与 Skill 输入要求匹配的资料。
4. 最近更新的历史需求和决策。
5. 竞品和外部参考。

输出结果需要展示引用来源，至少包括资料标题和引用摘要。

## Skill 执行链路

```text
用户输入
  ↓
默认通用 Skill：需求诊断、知识检索、缺口识别
  ↓
澄清问题或 Skill 推荐
  ↓
用户选择系统 Skill / 我的 Skill / 自动推荐 Skill
  ↓
默认通用 Skill 整理上下文
  ↓
目标 Skill 执行
  ↓
结果生成：方案、地图、PRD、流程、评审材料
  ↓
质量检查：事实来源、缺失信息、输出格式、验收标准
  ↓
沉淀为项目资产
```

## MVP 范围

### 必须包含

- 默认通用 Skill。
- 三个系统 Skill：模糊需求澄清、用户旅程地图、PRD/交互方案生成。
- Skill 中心基础列表。
- Skill 编辑器：表单模式和 Markdown 模式。
- 项目知识库基础导入和摘要。
- Skill 执行记录。
- 输出资产保存。
- 外部 Skill 导入为待审核草稿。

### 暂不包含

- Skill 市场。
- 多人协作权限。
- 自动发布到 Figma、Jira、Confluence。
- 完整插件运行时。
- 直接执行 GitHub 仓库中的代码。
- 高保真页面自动生成的完整闭环。

### 成功标准

- 用户输入模糊需求后，系统能给出需求诊断和推荐下一步。
- 用户能选择系统 Skill 或自己的 Skill 执行。
- 用户能通过表单或 Markdown 创建一个可运行的 Skill。
- 输出结果能引用知识库资料，并标注 AI 推断。
- 每次执行都能保存为项目资产和运行记录。

## 风险与开放问题

### 风险

- 如果知识库结构化质量低，Skill 输出仍会泛。
- 如果 Skill 编辑过于复杂，初级用户会放弃创建。
- 如果外部导入缺少审核，存在 prompt injection 和数据泄露风险。
- 如果内置 Skill 过多，用户会陷入选择困难。

### 应对

- MVP 只做三个内置 Skill，先验证执行链路。
- 默认入口不要求用户选 Skill，由默认通用 Skill 推荐。
- Skill 编辑先表单化，高级能力放在 Markdown。
- 外部 Skill 只读导入，默认待审核，不执行脚本。
- 输出强制区分知识库事实和 AI 推断。

## 需要用户确认

- MVP 是否先聚焦“默认通用 Skill + 三个系统 Skill + 自定义 Skill 编辑器”。
- 第一版知识库是否只做文本和网页资料，暂缓 Figma/代码深度解析。
- 输出资产是否先用 Markdown 形态沉淀，后续再扩展到 Figma、Jira、Confluence。
