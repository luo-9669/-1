# AI 工作流卡片式需求拆解交互设计

## 背景

当前流程运行页已经可以完成“选择流程 → 填字段 → 生成当前步骤 → 编辑输出 → 确认进入下一步”。这解决了可点击流程的问题，但还不够像真实的产品/UX 设计工作方式。

产品交互设计师使用 AI 时，不是一次生成就结束，而是会不断质疑、补充、要求 AI 重新考虑约束，再采纳一个更可靠的中间结论。需求拆解尤其如此：用户需要先看到 AI 对需求的理解，然后指出哪里不对，再让 AI 基于质疑重新生成，最后才确认进入下一步。

本阶段目标是把流程运行页升级为“AI 工作流卡片”，让每一步都成为可质疑、可重生成、可采纳、可沉淀的工作单元。

## 产品目标

- 用户输入一个原始需求后，可以进入卡片式流程。
- 每个步骤以一个独立卡片呈现，不再只是表单加 textarea。
- 每张卡片支持：生成结果、提出质疑、补充修改要求、重新生成、采纳本步、进入下一步。
- 每次重新生成都保留版本，用户可以回看和恢复。
- 下一步默认继承上一步采纳结果，而不是让用户从空白开始。
- 最终保存资产时包含每一步的采纳版本和关键质疑记录。

## 范围

### 本阶段包含

- 重构流程运行页的当前步骤区域。
- 新增每步的“质疑/修改意见”输入区。
- 新增“重新生成”动作。
- 新增步骤输出版本记录。
- 新增“采纳本步”状态。
- 让下一步生成时能读取前面已采纳输出。
- 导出 Markdown 时包含采纳输出和质疑记录。

### 本阶段不包含

- 真正接入大模型服务。
- 多人协作批注。
- 富文本编辑器。
- 可视化白板或流程图拖拽。
- Figma 自动出图。

## 推荐交互方案

采用“单页纵向卡片流 + 当前卡片聚焦”的方式。

页面顶部保留流程选择和原始需求输入。开始流程后，下方显示卡片流：

1. 当前步骤卡片展开。
2. 已采纳步骤卡片折叠为摘要。
3. 未解锁步骤只展示标题和目标。

这样用户既能看到完整流程，也不会被太多内容压住。

## 卡片结构

每张 AI 工作流卡片包含以下区域：

### 1. 步骤标题

展示：

- 步骤名称
- 步骤目标
- 状态：未开始、待输入、已生成、已质疑、已采纳、已锁定

### 2. 本步输入

展示本步骤需要的信息。

输入来源：

- 当前步骤必填字段
- 原始需求
- 前面步骤已采纳输出
- 用户补充信息

字段仍然保留结构化表单，但不再是唯一输入方式。

### 3. AI 产出

生成后展示当前版本输出。

输出应支持用户直接编辑。编辑后的内容可以作为采纳版本。

### 4. 质疑与修改

用户可以输入对当前产出的质疑或修改要求，例如：

- “目标用户不应该是管理员，而是审批发起人。”
- “补充异常流程和权限不足状态。”
- “这个结果太泛，请结合项目知识库重新拆。”
- “按 B 端后台页面来组织，不要按 C 端流程。”

系统提供快捷质疑按钮：

- `哪里不清楚？`
- `有什么风险？`
- `补异常场景`
- `更贴近 B 端`
- `重新结合知识库`

### 5. 操作区

主要操作：

- `生成本步`
- `重新生成`
- `采纳本步`
- `进入下一步`

规则：

- 第一次没有输出时显示 `生成本步`。
- 有输出且有质疑内容时显示 `重新生成`。
- 有输出时显示 `采纳本步`。
- 只有采纳后才允许进入下一步。

## 需求拆解推荐流程

需求拆解可以拆成 6 张卡片：

1. `需求理解`
   - 输出：当前理解、目标用户、业务目标、成功标准、缺失信息。

2. `需求质疑`
   - 输出：不清楚的问题、冲突点、风险、关键追问。

3. `场景拆解`
   - 输出：用户场景、主流程、异常流程、边界条件。

4. `方案生成`
   - 输出：MVP 方案、完整方案、低风险方案。

5. `交互转化`
   - 输出：页面结构、核心组件、关键状态、反馈机制。

6. `质量校验`
   - 输出：验收标准、风险清单、待确认问题、开发任务。

现有 `需求拆解四步法` 可以保留，但建议新增一个更适合交互设计师使用的 `需求拆解卡片流` workflow。

## 数据模型

现有 workflow run 增加以下字段：

```js
{
  stepOutputs: {
    [stepId]: '当前采纳输出'
  },
  stepDraftOutputs: {
    [stepId]: '当前正在编辑的输出'
  },
  stepChallenges: {
    [stepId]: [
      {
        id,
        content,
        createdAt
      }
    ]
  },
  stepVersions: {
    [stepId]: [
      {
        id,
        content,
        challenge,
        createdAt,
        accepted
      }
    ]
  }
}
```

含义：

- `stepDraftOutputs` 是当前卡片正在编辑或刚生成的内容。
- `stepOutputs` 只保存用户已采纳的内容。
- `stepChallenges` 保存用户对该步提出的质疑。
- `stepVersions` 保存每次生成和重新生成的结果。

## 生成逻辑

### 第一次生成

输入：

- 原始需求
- 当前步骤表单字段
- 前面已采纳步骤输出

输出：

- 当前步骤初版结果
- 生成一个 step version

### 重新生成

输入：

- 原始需求
- 当前步骤表单字段
- 当前输出
- 用户质疑/修改意见
- 前面已采纳步骤输出

输出：

- 新版本结果
- 记录质疑内容
- 生成新的 step version

### 采纳本步

动作：

- 将当前输出写入 `stepOutputs[stepId]`
- 将当前版本标记为 accepted
- 将步骤状态改为 accepted/completed
- 解锁下一步

## 页面状态

### 卡片状态

- `locked`：未解锁，只显示标题和目标。
- `active`：当前步骤，可输入和生成。
- `generated`：已有 AI 输出，待用户采纳或质疑。
- `challenged`：用户已提出质疑，可重新生成。
- `accepted`：用户已采纳，可进入下一步。

### 按钮状态

- 缺少必填字段：禁用 `生成本步`。
- 没有输出：禁用 `采纳本步`。
- 没有采纳：禁用 `进入下一步`。
- 没有质疑内容：禁用 `重新生成`，但允许用户直接编辑输出后采纳。

## 错误处理

- 必填字段缺失：显示缺失字段。
- 重新生成时质疑为空：提示用户先写修改意见。
- 输出为空：不能采纳。
- 进入下一步时未采纳：提示先采纳当前步骤。
- 版本恢复失败：保留当前输出并提示用户重新选择版本。

## 测试策略

### 单元测试

- 创建 workflow run 时包含 `stepDraftOutputs`、`stepChallenges`、`stepVersions`。
- 第一次生成会写入 draft output 和 version。
- 重新生成会读取 challenge 并新增 version。
- 采纳本步会写入 step output，并标记版本 accepted。
- 未采纳时不能进入下一步。
- Markdown 导出包含采纳输出和质疑记录。

### 浏览器验证

- 输入原始需求后进入卡片式流程。
- 当前卡片可以生成输出。
- 用户填写质疑内容后可以重新生成。
- 重新生成后版本数量增加。
- 点击采纳后才能进入下一步。
- 下一步能看到上一步已采纳摘要。
- 保存资产后资产详情包含质疑记录。

## 实施边界

- 本阶段优先改流程运行页，不改变项目诊断、资产库和 Skill 中心的大结构。
- 本阶段只做本地 deterministic 生成逻辑，不接真实 AI。
- 现有 `交互方案生成` workflow 继续可用。
- 新增 `需求拆解卡片流` workflow，避免破坏已有 `需求拆解四步法`。
